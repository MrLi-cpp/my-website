第零章：环境搭建——从0到1，让代码跑起来

本章目标：在你的电脑上安装 Node.js，创建一个项目文件夹，让浏览器能访问到你写的网页。不需要懂任何编程知识，跟着做就行。

学完后你能看到什么：打开浏览器，输入 localhost:3000，能看到一个写着"Hello World"的网页——这是你亲手搭建的第一个网站服务器。

0.1 为什么需要"环境"？——从打字到让全世界看见

如果你用过 Word 写文档，你肯定知道：写完文档后，双击文件就能用 Word 打开。但网页不一样。你写的 HTML 文件双击打开，浏览器地址栏显示的是 file:///Users/.../index.html——这是一个本地文件路径，不是网址。

这有什么问题？

问题在于：当你以后要"让全世界看见"的时候，你需要一个服务器来回答浏览器的请求。而服务器不会直接打开你电脑上的文件——它需要在一个特定的"运行环境"中工作。

类比一下：你写了一封信（HTML文件），想寄给别人看。你不能直接把信塞到对方手里（双击打开），你需要一个邮局（服务器）来帮你投递。而邮局需要雇员工、买设备才能运转——Node.js 就是这个"员工"，它懂得如何处理浏览器的请求。

📖 故事时间：Node.js 是怎么诞生的？

2009年，一个叫 Ryan Dahl 的程序员在旧金山的一个技术聚会上，展示了一个奇怪的项目。他说："JavaScript 只能在浏览器里运行，这太浪费了。"

当时的背景是：2008年 Google 发布了 Chrome 浏览器，里面的 V8 引擎让 JavaScript 运行速度快了几十倍。Ryan 想：既然 JavaScript 这么快，为什么不能让它在服务器上跑呢？

传统的服务器程序用 Java、PHP、Python 写，每次有新的用户请求，服务器就启动一个新的"线程"来处理。如果有几千个用户同时访问，服务器就要启动几千个线程——内存很快就耗尽了。Ryan 的想法是：JavaScript 的事件循环机制可以让一个线程同时处理成千上万个请求，就像餐厅里一个服务员同时照顾很多桌客人一样。

这个 idea 在当时被认为是疯狂的。JavaScript 是"玩具语言"，怎么能做服务器？但 Ryan 坚持做了出来，命名为 Node.js（Node 是节点的意思，暗示它是网络中的一个节点）。

2010年，Node.js 开始流行。2011年，LinkedIn 用 Node.js 重写了他们的移动应用后端，性能提升了 20 倍。2013年，沃尔玛在黑色星期五（一年中流量最大的一天）用 Node.js 处理了几亿次请求，没有崩溃。

今天，Node.js 已经是全栈开发的标准工具之一。我们用它来搭建网站后端，也用它来运行各种开发工具（比如你这个项目里的 npm）。

🤔 停下来想一想

Ryan Dahl 为什么要让 JavaScript 跑到服务器上？为什么不直接用当时已经成熟的 PHP 或 Java？

答案的关键在于"事件循环"和"非阻塞 I/O"。如果你同时有 1000 个用户，PHP 需要启动 1000 个进程；而 Node.js 只需要一个进程，在这个进程里不断切换处理不同的请求。就像一个人同时 juggling 很多球，而不是找 1000 个人各拿一个球。

这种设计让 Node.js 在处理高并发（同时很多请求）时非常高效。对于个人网站来说，并发量不大，但 Node.js 的简洁和生态丰富让它成为最佳选择。

0.2 安装 Node.js——给电脑请一个"服务员"

0.2.1 Node.js 是什么？

Node.js 不是一门新语言。它是一个运行环境——让 JavaScript 代码在你的电脑上运行的程序。你可以把它理解为"JavaScript 的解释器"。

通常我们写 JavaScript 是为了让网页有交互效果（比如点击按钮弹出提示），这种代码只能在浏览器里运行。Node.js 打破了这个限制：同样的 JavaScript 代码，现在可以在服务器上运行了。

📖 故事时间：npm 和"依赖地狱"

Node.js 刚出来的时候，安装第三方库非常麻烦。如果你想用一个处理日期的库，你需要手动下载文件、复制到项目里、解决各种兼容问题。如果有 A 库依赖 B 库，B 库又依赖 C 库……你就要手动下载 A、B、C、D、E……这叫"依赖地狱"。

2010年，npm（Node Package Manager）发布了。它的灵感来自 Linux 世界的包管理器（比如 apt、yum）。npm 的想法很简单：建立一个中央仓库，所有人把自己写的代码传上去。当你需要一个库时，只需运行 npm install 库名，npm 会自动下载这个库，以及它依赖的所有其他库，全部放到你的项目里。

今天，npm 仓库里有超过 200 万个包。从处理图片、连接数据库到发送邮件，几乎所有你能想到的功能都有人写好了。这就是为什么现代开发如此高效——你不需要从零造轮子，只需 npm install。

但 npm 也带来了"依赖膨胀"的问题。一个项目可能依赖了几百个包，每个包又依赖几百个包……最终 node_modules 文件夹可能有几十万个文件。2016年有个著名的笑话：一位程序员不小心删除了 node_modules 文件夹，然后他的项目就永远消失了——因为重新安装需要太长时间，而他已经辞职了。

0.2.2 安装步骤

步骤1：打开官网
访问 https://nodejs.org，下载 LTS（长期支持）版本。LTS 意味着这个版本会维护几年，不会频繁更新导致兼容性问题。
步骤2：安装
双击下载的 .pkg 文件，一路点"继续"即可。Mac 可能会提示需要输入密码，正常输入就行。
步骤3：验证安装
打开 Terminal（终端），输入： 
node --version

如果显示类似 v20.11.0 的版本号，说明安装成功。

再输入：

npm --version

如果显示类似 10.2.3 的版本号，说明 npm 也安装好了。

💡 命令行是什么？
Terminal 是一个"文本界面的电脑"。平时你用鼠标点图标、拖窗口，这是图形界面。Terminal 里你输入文字命令来让电脑做事。它看起来简陋，但对程序员来说反而更高效——因为鼠标能做的事有限，文字命令可以做到任何事。

🤔 为什么要用命令行安装？

你可能会问：为什么不是像安装微信一样，双击图标就完事了？

因为开发工具和面向普通用户的软件设计理念不同。微信要照顾所有人，所以界面尽可能简单。而开发工具面向的是"愿意学习的人"，命令行提供了最大的灵活性和自动化能力。

更重要的是：命令行可以被脚本调用。你可以写一个文件，里面包含 100 个命令，然后一键执行。这在图形界面里几乎不可能做到。这种"可脚本化"是现代软件开发的基础。

0.3 创建项目——你的第一个代码家

0.3.1 为什么是 "my-website"？

项目名字可以随便起，但 "my-website" 这个名字很直白——"我的网站"。以后你可能会做很多项目，起一个有描述性的名字会让你一眼就知道这个项目是干什么的。

程序员有个坏习惯：起名字太随意。比如 "test"、"project1"、"aaa"。这些名字在一个月后就完全不知道是什么了。霁光这个项目名字就很好——"霁"是雨后天晴的意思，"光"是光明。网站就像雨后的阳光，记录成长、分享思考。

步骤1：创建文件夹
在 Terminal 中输入： 
mkdir ~/Desktop/my-website

mkdir 是 "make directory" 的缩写，意思是"创建文件夹"。~/Desktop/ 是你的桌面路径。~ 在 Unix 系统中代表"用户主目录"，对霁光来说就是 /Users/mrli/。

步骤2：进入文件夹
cd ~/Desktop/my-website

cd 是 "change directory" 的缩写。现在你的 Terminal 就"站"在这个文件夹里了，后续的所有命令都会在这个文件夹里执行。

步骤3：初始化项目
npm init -y

这个命令做了什么事？

npm init 会创建一个 package.json 文件。这个文件是项目的"身份证"——里面记录了项目的名字、版本、作者、依赖的第三方包等信息。没有它，npm 不知道该怎么管理你的项目。

-y 参数表示"全部用默认值"，不用一个一个回答提示问题。

执行后，你会看到文件夹里多了一个 package.json，内容大概是：

{
  "name": "my-website",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}

这些字段的含义：

字段含义
name项目名字。npm 安装依赖时，这个名字作为文件夹名的一部分。
version版本号，格式是"主版本.次版本.修订号"。
main入口文件。当别人 import 你的项目时，会加载这个文件。
scripts命令脚本。比如 "start": "node server.js"，以后运行 npm start 就能启动服务器。
author作者名。可以写 "霁光" 或你的名字。

🤔 为什么需要 package.json？

你可能会想：我直接写代码不行吗？为什么要一个配置文件？

想象你在做一个乐高模型。package.json 就是"说明书"——它告诉你这个模型需要哪些零件（依赖）、怎么组装（scripts）、是谁设计的（author）。当你把项目分享给朋友时，他不需要把所有零件一起打包，只需要这个说明书和一小袋"特殊零件"，然后按说明书去商店买通用零件就行。

在现代软件开发中，"分享"是常态。你把代码推送到 GitHub，其他人 clone 下来，运行 npm install，就能复现你的环境。没有 package.json，这就像没有说明书的乐高——别人完全不知道该怎么拼。

0.4 安装 Express——站在巨人的肩膀上

0.4.1 什么是 Express？

Express 是 Node.js 最流行的 Web 框架。"框架"是什么意思？想象你要盖房子。你可以从零开始一砖一瓦地砌（直接用 Node.js 内置的 http 模块），也可以买一套预制板房（Express），它已经把地基、框架、门窗都搭好了，你只需要往里面填内容。

Express 做了哪些"预制"工作？

  • 帮你监听端口（浏览器请求从哪个门进来）
  • 帮你解析请求（浏览器说了什么）
  • 帮你组织响应（怎么回答浏览器）
  • 帮你托管静态文件（CSS、JS、图片怎么发给浏览器）

📖 故事时间：Express 的诞生

2010年，TJ Holowaychuk 发布了一个叫 Express.js 的项目。TJ 是一位极其高产的开源作者——他创建了 Express、Jade（后来的 Pug）、Mocha（测试框架）、Coffeescript 编译器等几十个流行项目。

Express 的设计哲学是"极简"——它只做 Web 服务器最核心的功能，其他功能通过"中间件"插件来扩展。这种设计让 Express 非常轻量（核心代码只有几千行），同时也极其灵活。

2014年，TJ 做了一件震惊开源社区的事：他把 Express 以及他维护的所有项目的所有权全部捐赠给了 StrongLoop 公司，然后宣布退出 Node.js 社区，转而学习 Go 语言。他说："我对 JavaScript 生态感到疲惫。"

这个故事告诉我们：开源世界没有永恒的主人。Express 后来几经转手（StrongLoop → IBM → OpenJS Foundation），但代码一直在运行，全世界的网站都在用它。这就是开源的力量——代码属于所有人。

0.4.2 安装 Express

步骤
确保你在 my-website 文件夹里，然后输入： 
npm install express

npm 会联网下载 express 包，以及 express 依赖的所有其他包，放到 node_modules 文件夹里。同时 package.json 里会多出一个字段：

"dependencies": {
  "express": "^4.18.2"
}

^4.18.2 是什么意思？

这是语义化版本号（Semantic Versioning，简称 SemVer）。格式是 主版本.次版本.修订号：

部分含义何时变化
主版本（4）大版本不兼容的 API 改动
次版本（18）功能版本新增功能，但向下兼容
修订号（2）补丁版本修复 Bug，不改功能

^ 符号表示"允许次版本和修订号更新，但主版本不变"。所以 ^4.18.2 会匹配 4.18.2、4.19.0、4.20.5……但不会匹配 5.0.0（因为主版本变了，可能不兼容）。

🤔 为什么版本号这么重要？

2016年，一个名叫 Azer Koçulu 的程序员因为 npm 包命名纠纷，一怒之下从 npm 删除了他维护的所有包。其中有一个叫 "left-pad" 的包，功能非常简单：给字符串左边补空格。但全世界有成千上万个项目依赖这个包（包括 Facebook、Babel 这样的大项目）。left-pad 被删除后，这些项目在重新安装依赖时全部报错，半个互联网瘫痪了几个小时。

这个故事说明了两件事：第一，现代软件依赖链非常脆弱；第二，版本号管理至关重要。如果 npm 允许随意覆盖已有版本，那么"今天能用的代码"可能在明天因为某个依赖的更新而崩溃。SemVer 和 package-lock.json（记录精确版本号）就是为了解决这个问题。

0.5 写第一个服务器——Hello World 的仪式感

0.5.1 创建 server.js

步骤1：创建文件
在 my-website 文件夹里创建 server.js。

怎么创建？在 Terminal 里输入：

touch server.js

touch 命令创建一个空文件。然后用编辑器打开它（VS Code 可以直接点 server.js 文件编辑）。

步骤2：写入代码
// 文件位置：my-website/server.js

// 第1行：引入 express
// require 是 Node.js 的模块加载系统
// 它去 node_modules 文件夹里找到 express 包，加载进来
const express = require('express');

// 第2行：创建应用实例
// app 就是一个"空的"服务器，还没配置任何功能
const app = express();

// 第3行：定义路由
// 当浏览器访问 localhost:3000 时，返回 "Hello World!"
// '/' 是"根路径"，就是域名后面什么都不加
app.get('/', (req, res) => {
  res.send('Hello World!');
});

// 第4行：启动服务器
// 监听 3000 端口，启动后打印一条消息
app.listen(3000, () => {
  console.log('服务器已启动！访问 http://localhost:3000');
});

0.5.2 逐行解剖

第1行：const express = require('express');

require 是 Node.js 的模块系统。当 Node.js 执行到这行时，它会：

  1. 先检查有没有 express.js 或 express/index.js 这样的本地文件——没有
  2. 然后去 node_modules 文件夹里找名为 express 的包——找到了
  3. 加载 express 包的入口文件（就是 express 自己的 package.json 里 main 字段指定的文件）
  4. 把加载结果赋给 express 变量

你可能好奇：为什么不是 require('./express') 而是 require('express')？

区别就在于前面有没有 ./：

  • require('./xxx')：找当前文件夹下的 xxx 文件（本地文件）
  • require('xxx')：去 node_modules 里找 xxx 包（第三方包）

📖 故事时间：为什么叫 require？

Node.js 的模块系统灵感来自 Ruby 的 require 和 Python 的 import。CommonJS 规范（就是 Node.js 遵循的模块规范）制定者想要一个简洁的函数名来表达"我需要这个东西"。

但 CommonJS 不是唯一的模块规范。2015年，ES6 发布了官方模块语法：import express from 'express'。这种语法更简洁，但 Node.js 直到 13.2.0 版本才正式支持（而且需要把文件后缀改成 .mjs，或者在 package.json 里加 "type": "module"）。

这就是为什么你现在看到的大多数 Node.js 项目仍然用 require——它兼容所有版本，而 import 还比较新。不过 Express 的下一个大版本（Express 5）可能会全面转向 ES 模块。

第2行：const app = express();

express() 是一个函数调用。它返回一个"应用实例"——你可以把它理解为一台还没通电的 Web 服务器。这时候它什么都不会做，你需要往上面"挂载"各种功能。

为什么设计成这样？因为 Express 遵循"可组合"的设计理念。你可以创建多个 app 实例，每个实例有不同的配置，然后把它们组合在一起。虽然对小项目来说一个实例就够了，但这种设计让 Express 在大项目里非常灵活。

第3行：app.get('/', (req, res) => { ... });

这是整个服务器最核心的部分——路由。

什么是路由？想象你走进一家餐厅：

  • 你说"我要一份牛排" → 服务员告诉厨房做牛排 → 这就是路由匹配
  • 你说"我要一杯水" → 服务员给你倒水 → 另一条路由
  • 你说"我要一艘航母" → 服务员不知道该怎么办 → 404 未找到

在 Web 服务器里，URL 就是"点餐内容"，路由处理函数就是"厨房"。

app.get('/', handler) 表示：当浏览器用 GET 方法访问根路径 / 时，执行 handler 函数。

handler 函数有两个参数：

  • req（request）：请求对象。里面包含了浏览器发来的所有信息——URL、请求头、请求体、Cookie 等等。
  • res（response）：响应对象。用来"回复"浏览器。你可以设置状态码、响应头、响应体。

res.send('Hello World!') 表示：把字符串 "Hello World!" 作为响应体发回浏览器，并自动设置 Content-Type 为 text/html。

🤔 为什么 req 和 res 的顺序不能反过来？

Express 规定处理函数的参数顺序必须是 (req, res, next)。这是约定俗成的设计，几乎所有 Node.js Web 框架都遵循这个顺序。为什么不是 (res, req)？

一种说法是：HTTP 请求是"先读后写"的过程——服务器先读请求（req），再写响应（res）。所以 req 在前。另一种说法是：这个设计继承自 Ruby 的 Sinatra 框架，而 Sinatra 又继承自更早的约定。

无论原因如何，记住这个顺序就好。如果你写反了，代码不会报错，但你的 req 变量实际上指向响应对象，res 变量指向请求对象——这会导致非常诡异的 Bug。

第4行：app.listen(3000, callback);

这行代码启动服务器，让它"开机运行"。

3000 是端口号。端口是什么？

想象你的电脑是一栋大楼，每个程序是一个住户。浏览器要访问你的服务器，需要知道"哪个门"。端口号就是这个"门牌号"。

常见的端口号：

端口用途
80HTTP 默认端口
443HTTPS 默认端口
3000开发常用（Express、React 默认）
8080备选 HTTP 端口
22SSH 远程登录
21FTP 文件传输

为什么开发常用 3000？这其实没有严格的规定，只是一个约定俗成的习惯。Ruby on Rails 默认用 3000，Express 也跟着用了 3000，然后 React 开发服务器也用 3000……于是 3000 就成了"Web 开发"的代名词。

你完全可以改成 8080、8888、甚至 12345——只要这个端口没有被其他程序占用。

📖 故事时间：localhost 是谁？

localhost 是一个特殊的域名，它永远指向你自己的电脑。在技术实现上，localhost 被解析为 IP 地址 127.0.0.1。

127.0.0.1 属于"回环地址"（loopback address）范围。这个范围内的所有地址（127.0.0.0 到 127.255.255.255）都不会发到网络上，而是直接在操作系统内部"转个圈"回来。就像你给住在同一个房间的人写信，信不出房门就递到了。

这种设计有几个好处：第一，测试时不需要网络；第二，外部用户无法访问 127.0.0.1，天然安全；第三，速度极快（不需要经过网卡）。

那为什么用 127.0.0.1 而不是 1.1.1.1？这要追溯到 1981 年的 RFC 790 文档。当时互联网先驱们把 127 开头的整个 A 类地址段预留给回环测试，而 127.0.0.1 是最"规整"的第一个地址。从此它就成了 localhost 的默认映射。

0.6 托管 HTML 文件——让服务器"展示"网页

0.6.1 为什么需要托管？

刚才的 Hello World 是直接由服务器生成的字符串。但真实的网站有 HTML 文件、CSS 样式表、JavaScript 脚本、图片……这些文件不能都写成字符串塞在 server.js 里。

我们需要告诉服务器："有一个文件夹叫 public，里面的所有文件都直接发给浏览器。"

这叫静态文件托管（static file serving）。"静态"的意思是这些文件内容固定，不需要服务器"思考"——浏览器要什么，服务器就原样给什么。

📖 故事时间：HTTP 协议的发明

1989年，在瑞士 CERN（欧洲核子研究中心）工作的英国科学家 Tim Berners-Lee 遇到了一个难题：物理学家们分布在全球各地，他们的研究报告、实验数据、软件工具散落在不同的电脑上，格式各异，很难找到。

Tim 想：如果有一种统一的方式，让任何人在任何电脑上都能访问任何文档，该多好？

他设计了三个东西：

  1. HTML（超文本标记语言）：一种统一的文档格式
  2. HTTP（超文本传输协议）：一种请求/回复的通信规则
  3. URL（统一资源定位符）：一种标识文档位置的地址格式

1990年，Tim 写出了世界上第一个 Web 服务器（httpd）和第一个浏览器（WorldWideWeb，后来改名 Nexus）。1991年，第一个网站上线了——它就在 CERN 的一台 NeXT 电脑上。

Tim 本可以通过专利成为亿万富翁，但他选择了把 Web 技术免费开放给所有人。他说："Web 的意义在于连接全人类，而不是赚钱。"今天，Tim 仍然在推动互联网的开放标准，他创办的 W3C（万维网联盟）制定了 HTML、CSS、HTTP 等所有 Web 核心规范。

HTTP 的最初版本极其简单——只有 GET 方法，请求就是一个单词 "GET" 加路径，响应就是原始 HTML。1996 年的 HTTP/1.0 增加了状态码、请求头。1999 年的 HTTP/1.1 增加了持久连接、Host 头。2015 年的 HTTP/2 增加了多路复用。2022 年的 HTTP/3 基于 QUIC 协议，更快更安全。

我们用的 Express 的 res.send()、app.use(express.static())，底层都是 HTTP 协议。每次浏览器和服务器对话，都在遵循 Tim Berners-Lee 34 年前设计的规则。

0.6.4 CSS 代码详解——为什么这样设计？

box-sizing: border-box;

这是 CSS 历史上最重要的修复之一。

默认情况下（box-sizing: content-box），元素的 width 只包含内容区域。如果你给元素加了 padding 或 border，它的实际宽度会超出你设置的 width。这非常反直觉。

比如：

/* 默认模式 */
div {
  width: 200px;
  padding: 20px;
  border: 1px solid black;
  /* 实际占用宽度 = 200 + 20*2 + 1*2 = 242px ! */
}

设计师 Paul Irish 在 2012 年写了一篇著名的博客《*{box-sizing: border-box} FTW》（FTW = For The Win），建议所有人都用这个设置。border-box 让 width 包含内容 + padding + border，符合人类直觉。

今天，几乎所有 CSS 框架（Bootstrap、Tailwind、Normalize.css）都默认使用 border-box。它已经被拟议加入 CSS 标准作为默认值（虽然还没实现）。

🤔 为什么 CSS 默认不是 border-box？

这是一个典型的"历史遗留问题"。CSS 最早由 Håkon Wium Lie 在 1994 年提出，当时的设计理念深受 SGML（标准通用标记语言）和 DSSSL（文档样式语义和规范语言）影响。在那个时代，"内容区域"被视为核心，padding 和 border 被视为"附加装饰"，不应该计入尺寸。

但这个设计在实际使用中被证明是错误的——设计师们更关心"这个盒子占多少空间"，而不是"内容区有多宽"。于是 border-box 诞生了，并在 2012 年通过 Paul Irish 的博客大规模普及。

这也说明了一个道理：技术设计有时候会被实际使用场景"纠正"。理论上的完美不一定等于实践中的好用。

display: flex;

Flexbox（弹性盒子）是 CSS 布局的革命性进步。在 Flexbox 出现之前（2012年之前），网页布局主要靠 float（浮动）和 position（定位）。float 原本是为了让文字环绕图片设计的，后来被程序员"滥用"来做布局，导致无数 Bug 和 hacks。

Flexbox 的设计目标是"一维布局"——让元素在一条轴线上（水平或垂直）灵活排列。我们的 .home-wrap 用 flex-direction: column 让子元素垂直排列，用 justify-content: center 让它们在主轴上居中，用 align-items: center 让它们在交叉轴上居中。

我们的 .home-entries 用 flex-direction: row（默认值）让三个卡片水平排列，用 gap: 20px 控制间距。如果是老式的 float 布局，你需要给每个卡片加 margin，还要处理最后一个卡片的 margin-right 问题——非常繁琐。

📖 故事时间：CSS 从混乱到现代布局

1996年，CSS 1.0 发布时，float 是用来实现杂志式"文字环绕图片"效果的。但 Web 设计师很快发现：用 float 可以把块级元素并排显示，从而实现多列布局。于是 float 被"误用"了十几年。

这个误用带来了很多痛苦。比如"清除浮动"（clearfix）——当父元素只包含浮动的子元素时，父元素的高度会塌陷为 0。解决方案是一个 hack：在父元素末尾加一个看不见的 ::after 伪元素，设成 clear: both。

2009年，Flexbox 开始被浏览器实现。2012年，"新版"Flexbox 语法（也就是我们今天用的语法）稳定下来。2016年，Grid 布局（二维布局）加入 CSS。至此，CSS 终于有了健全的布局系统：

  • 一维布局（单行或单列）→ 用 Flexbox
  • 二维布局（网格）→ 用 Grid
  • 精确定位（覆盖层、弹窗）→ 用 Position

float 终于回归了它的本职工作——文字环绕图片。这是 CSS 发展史上最漫长的"拨乱反正"。

transition: all 0.3s ease;

这行代码让元素的所有属性变化都有动画效果。

当鼠标悬浮在卡片上时，transform: scale(1.08) 让卡片放大 8%。如果没有 transition，这个放大是瞬间完成的，看起来会很生硬。有了 transition，放大过程会在 0.3 秒内平滑完成。

ease 是时间函数，表示"先快后慢"。你可以想象一个弹簧：释放时速度最快，接近目标时逐渐减速。这是自然界中最常见的运动模式，所以看起来最舒服。

0.6.5 修改 server.js 托管静态文件

步骤4：修改 server.js
在现有代码基础上添加一行： 
// 文件位置：my-website/server.js

const express = require('express');
const app = express();

// ===== 新增：托管 public 文件夹 =====
// express.static 是一个"中间件"
// 它的作用是：当浏览器请求 CSS/JS/图片等静态文件时，
// 去 public 文件夹里找对应的文件，找到就发送，找不到就继续后面的路由
app.use(express.static('public'));

app.get('/', (req, res) => {
  res.send('Hello World!');
});

app.listen(3000, () => {
  console.log('服务器已启动！访问 http://localhost:3000');
});

中间件是什么？

想象你去餐厅吃饭，中间件就是"服务流程"：

  1. 进门 → 迎宾（中间件1）
  2. 点餐 → 服务员记录（中间件2）
  3. 厨房做菜 → 厨师处理（中间件3，路由处理函数）
  4. 上菜 → 服务员端菜（中间件4）
  5. 结账 → 收银员算账（中间件5）

每个中间件都可以"拦截"请求——如果迎宾发现你是黑名单，可以直接拒绝进入，后面的流程都不会执行。

express.static('public') 就是一个中间件。它检查：浏览器请求的 URL 是否对应 public 文件夹里的某个文件？如果是，直接发送文件，然后结束请求。如果不是，放行给下一个中间件或路由处理函数。

🤔 为什么 static 要在路由之前？

Express 的中间件和路由是"从上到下"依次匹配的。如果浏览器访问 /style.css：

如果 static 放在前面 → 发现 public 里有 style.css → 发送文件 → 结束。

如果 static 放在后面 → 先匹配到 app.get('/') → 不匹配（因为是 /style.css 不是 /）→ 继续 → 匹配到 static → 发送文件。

在这个特定例子里，顺序其实无所谓，因为 app.get('/') 只匹配根路径。但如果你的路由是 app.get('/:id')（匹配任何路径），那 static 放在后面就会导致静态文件请求被路由拦截。

最佳实践：static 总是放在最前面。

📖 故事时间：为什么浏览器要缓存？

1990年代的网络非常慢。一张 100KB 的图片可能需要下载几十秒。如果每次刷新页面都重新下载所有文件，体验会很差。

于是浏览器引入了缓存机制：第一次访问时下载文件并保存到本地磁盘，再次访问时检查"这个文件有没有更新？"如果没有，直接用本地保存的版本。

怎么知道有没有更新？服务器在响应头里发送两个字段：

  • Last-Modified：文件最后修改时间
  • ETag：文件的"指纹"（哈希值）

浏览器下次请求时带上这些信息，服务器比对后说"没变"（返回 304 Not Modified），浏览器就加载本地缓存。

这对开发来说是个麻烦——你修改了 CSS，但浏览器还在用旧的。解决方案：

  • 按 Ctrl+F5 强制刷新（跳过缓存）
  • 在 DevTools 的 Network 标签勾选 Disable Cache
  • 给文件 URL 加随机参数 ?v=123（每次修改改数字）

生产环境中，缓存是性能优化的关键。一个优化良好的网站，90% 的请求都是 304，不需要重新下载。

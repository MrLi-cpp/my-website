第十章：私信系统——点对点的消息交流

本章目标：实现两个用户之间的私信功能。可以发送消息、查看对话列表、实时刷新。

学完后你能做什么：有一个 /chat.html 页面，选择用户后开始对话，消息按时间顺序显示。你能理解轮询机制和消息存储设计。

10.1 为什么不用 WebSocket？

真正的实时聊天（微信、QQ）用 WebSocket——浏览器和服务器建立一个持久连接，服务器可以主动推送消息。但 WebSocket 需要额外的库（比如 socket.io），配置更复杂。

我们的方案是"轮询"（Polling）：前端每隔几秒请求一次新消息。虽然不够实时（有延迟），但实现简单，对个人网站足够用。

10.1.1 实时通信技术对比

技术实时性复杂度适用场景
轮询（Polling）低（秒级延迟）低低频消息、个人网站
长轮询（Long Polling）中中中等实时需求
Server-Sent Events高中服务器单向推送
WebSocket极高（毫秒级）高高频双向通信

📖 故事时间：WebSocket 的诞生

2008 年，Michael Carter 和 Ian Hickson 开始讨论"TCP for Web"——一种让浏览器和服务器建立双向通信通道的技术。当时实时通信主要靠轮询和长轮询，效率极低。

2011 年，WebSocket 被正式标准化（RFC 6455）。它的核心特点是：一次 HTTP 握手后建立持久 TCP 连接，之后双方可以随时发送数据，不需要重复建立 HTTP 连接。

WebSocket 的协议帧极小（2-14 字节头部），比 HTTP 的数百字节头部高效得多。今天的实时应用（在线游戏、股票行情、协作编辑）都用 WebSocket。

但 WebSocket 也有缺点：需要额外维护连接状态、心跳检测、断线重连。对于"偶尔发几条消息"的场景，轮询更简单可靠。

2021 年，WebTransport 作为 WebSocket 的继任者开始标准化。它基于 HTTP/3 和 QUIC 协议，支持多路复用、流控制、更好的拥塞控制。但 WebSocket 在短期内不会被淘汰——它已经被广泛部署和使用。

10.2 数据库设计——conversations 和 messages

私信需要两张表：对话（conversations）和消息（messages）。

为什么需要 conversations 表？为什么不能直接在 messages 里存 sender_id 和 receiver_id？

因为：1）两个用户之间的所有消息应该被分组（一个对话包含多条消息）；2）对话有元数据（创建时间、最后活动时间）；3）方便查询"我和谁聊过天"。

CREATE TABLE IF NOT EXISTS conversations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user1_id INTEGER NOT NULL,
  user2_id INTEGER NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(user1_id, user2_id)
);

CREATE TABLE IF NOT EXISTS messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  conversation_id INTEGER NOT NULL,
  sender_id INTEGER NOT NULL,
  content TEXT NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

conversations 表的 UNIQUE(user1_id, user2_id) 确保两个用户之间只有一个对话。

🤔 为什么 user1_id 和 user2_id 不区分"发起者"和"接收者"？

对话是对称的——没有"谁发起谁接收"的区别。但为了 UNIQUE 约束生效，我们需要一个规则来确保 (A,B) 和 (B,A) 被视为同一个对话。

解决方案：存储时总是把较小的 user_id 放在 user1_id，较大的放在 user2_id。这样无论 A 找 B 还是 B 找 A，存储的都是 (min(A,B), max(A,B))。

这类似于数学中的"无序对"概念——{A,B} 和 {B,A} 是同一个集合。

10.3 后端 API

10.3.1 获取或创建对话

app.post('/api/conversations', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  const { targetUserId } = req.body;
  const userId = req.session.userId;
  
  // 确保 user1_id < user2_id，避免重复
  const [u1, u2] = userId < targetUserId ? [userId, targetUserId] : [targetUserId, userId];
  
  db.get(
    'SELECT * FROM conversations WHERE user1_id = ? AND user2_id = ?',
    [u1, u2],
    (err, conv) => {
      if (err) return res.status(500).json({ error: err.message });
      
      if (conv) {
        return res.json(conv);
      }
      
      db.run(
        'INSERT INTO conversations (user1_id, user2_id) VALUES (?, ?)',
        [u1, u2],
        function(err2) {
          if (err2) return res.status(500).json({ error: err2.message });
          res.json({ id: this.lastID, user1_id: u1, user2_id: u2 });
        }
      );
    }
  );
});

10.3.2 发送消息

app.post('/api/conversations/:id/messages', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  const { content } = req.body;
  const convId = req.params.id;
  const senderId = req.session.userId;
  
  if (!content || content.trim() === '') {
    return res.status(400).json({ error: '消息不能为空' });
  }
  
  db.run(
    'INSERT INTO messages (conversation_id, sender_id, content) VALUES (?, ?, ?)',
    [convId, senderId, content],
    function(err) {
      if (err) return res.status(500).json({ error: err.message });
      res.json({ 
        id: this.lastID, 
        conversation_id: convId, 
        sender_id: senderId, 
        content,
        created_at: new Date().toISOString()
      });
    }
  );
});

10.3.3 获取消息

app.get('/api/conversations/:id/messages', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  db.all(
    'SELECT * FROM messages WHERE conversation_id = ? ORDER BY created_at ASC',
    [req.params.id],
    (err, rows) => {
      if (err) return res.status(500).json({ error: err.message });
      res.json(rows);
    }
  );
});

10.3.4 获取对话列表

app.get('/api/conversations', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  const userId = req.session.userId;
  
  db.all(
    `SELECT c.*, 
      CASE WHEN c.user1_id = ? THEN c.user2_id ELSE c.user1_id END as other_user_id,
      u.username as other_username
    FROM conversations c
    JOIN users u ON u.id = CASE WHEN c.user1_id = ? THEN c.user2_id ELSE c.user1_id END
    WHERE c.user1_id = ? OR c.user2_id = ?
    ORDER BY c.created_at DESC`,
    [userId, userId, userId, userId],
    (err, rows) => {
      if (err) return res.status(500).json({ error: err.message });
      res.json(rows);
    }
  );
});

这个查询用 CASE WHEN 来找出"对方"的用户 ID，然后 JOIN users 获取用户名。

10.5 轮询的实现

// 每 3 秒刷新一次消息
let pollInterval;

function startPolling(conversationId) {
  // 先立即获取一次
  loadMessages(conversationId);
  
  // 然后每 3 秒轮询
  pollInterval = setInterval(() => {
    loadMessages(conversationId);
  }, 3000);
}

function stopPolling() {
  if (pollInterval) {
    clearInterval(pollInterval);
    pollInterval = null;
  }
}

async function loadMessages(conversationId) {
  const res = await fetch(`/api/conversations/${conversationId}/messages`);
  const messages = await res.json();
  
  renderMessages(messages);
  
  // 滚动到底部
  const container = document.getElementById('chat-messages');
  container.scrollTop = container.scrollHeight;
}

// 页面离开时停止轮询
window.addEventListener('beforeunload', stopPolling);

🤔 轮询的优化方向

轮询虽然简单，但有几个问题：

  • 无效请求：大部分时间没有新消息，但前端仍然在请求
  • 服务器压力：N 个用户 × 每 3 秒一次 = 大量请求
  • 消息延迟：最坏情况下，消息发送后要等 3 秒才显示

优化方案：

  • 指数退避：没有新消息时，逐渐增加轮询间隔（3s → 5s → 10s）
  • 最后消息 ID：只获取比上次最后一条更新的消息（?after_id=123）
  • 长轮询：服务器 hold 住请求，有新消息时才返回
  • WebSocket：终极方案，但复杂度更高

对于个人网站，简单轮询完全够用。当用户量增长时，再考虑升级。

10.6 知识点回顾

📚 本章核心概念回顾

  • WebSocket：双向持久连接，适合高频实时通信。
  • 轮询（Polling）：定时请求新数据，实现简单但不够实时。
  • conversations 表：存储对话关系，UNIQUE(user1_id, user2_id) 防止重复。
  • messages 表：存储具体消息内容，通过 conversation_id 关联。
  • CASE WHEN：SQL 条件表达式，用于动态选择列值。
  • 消息气泡：CSS flex 布局 + border-radius 不对称实现对话效果。
  • setInterval / clearInterval：JavaScript 定时器，用于轮询。
  • beforeunload：页面关闭前触发的事件，用于清理资源。

10.7 常见问题急救站

问题现象原因解决方案
消息重复 同一条消息出现多次 轮询获取了重复消息 记录最后获取的消息 ID，只获取比它新的
不能和对方聊天 提示对话不存在 targetUserId 传错了 确保 targetUserId 是对方用户的 id（不是 username）
消息没显示 发了消息但看不到 轮询间隔太长或请求失败 缩短轮询间隔；检查网络请求是否 200
页面卡顿 聊天页越来越卡 setInterval 积累过多，或 DOM 节点过多 离开页面时 clearInterval；限制显示的消息数量
发送失败 点击发送但消息没保存 conversation_id 错误或 content 为空 检查对话 ID；检查内容是否为空字符串
消息顺序错乱 消息不是按时间顺序 ORDER BY 条件错误或时区问题 确保 ORDER BY created_at ASC；使用 ISO 格式时间

第九章：管理员后台——权限管理与数据统计

本章目标：实现管理员后台。只有管理员能进入，可以查看统计数据、管理文章和动态。你将理解 RBAC 权限模型、数据统计查询和后台界面设计。

学完后你能做什么：有一个 /admin.html 页面，管理员登录后可以看到：用户数量、文章数量、动态数量、最新内容列表。可以删除违规内容。

9.1 为什么需要管理员后台？

任何有用户生成内容的系统都需要管理功能：删除违规内容、查看统计数据、管理用户权限。如果没有后台，这些内容就只能通过直接操作数据库来管理——既不安全也不方便。

9.1.1 权限管理的演进

计算机系统的权限管理经历了几个阶段：

第一阶段（1960s）：无权限区分。 早期多用户系统没有权限概念，所有用户都能访问所有文件。这导致了严重的安全问题——学生可以删除教授的研究数据。

第二阶段（1970s）：UNIX 权限模型。 UNIX 引入了"用户-组-其他"的三层权限（rwx），用数字表示（如 755）。这个模型简单有效，但粒度太粗——只能控制"读/写/执行"，无法做更细的控制。

第三阶段（1990s）：ACL（访问控制列表）。 每个资源有一个列表，记录谁能访问。ACL 比 UNIX 权限更灵活，但管理复杂——1000 个文件 × 100 个用户 = 10 万条权限记录。

第四阶段（2000s+）：RBAC（基于角色的访问控制）。 用户有角色，角色有权限。用户通过角色获得权限。这是今天最流行的权限模型。

📖 故事时间：RBAC 的诞生

1992 年，美国国家标准与技术研究院（NIST）的研究员 David Ferraiolo 和 Richard Kuhn 提出了 RBAC 模型。他们的动机很简单：美国政府机构有数百万员工，每个人需要访问不同的系统和数据。用 ACL 管理是不可能的。

RBAC 的核心洞察是："权限应该分配给角色，而不是个人。"一个人可能有多个角色（如"医生"+"研究员"），一个角色可以被多个人共享。这大大减少了权限管理的复杂度。

2004 年，NIST 发布了 RBAC 标准（ANSI/INCITS 359）。今天，RBAC 是几乎所有企业系统的标准权限模型。Windows Active Directory、AWS IAM、Kubernetes RBAC 都是基于这个模型。

RBAC 有三种级别：

  • RBAC0（核心）：用户-角色-权限的基本映射
  • RBAC1（层级）：角色可以继承（如"医生"继承"员工"的所有权限）
  • RBAC2（约束）：角色互斥（如一个人不能同时是"会计"和"审计"）

我们的项目用最简单的 RBAC0：两种角色——普通用户（user）和管理员（admin）。

9.2 添加管理员字段

在 users 表中添加 is_admin 字段：

ALTER TABLE users ADD COLUMN is_admin INTEGER DEFAULT 0;

SQLite 的 ALTER TABLE 能力有限，不支持某些修改。如果报错，可以：

  1. 删除 database.sqlite 重新创建（开发环境）
  2. 或者用创建新表→迁移数据→重命名的方法（生产环境）

设置管理员账号：

UPDATE users SET is_admin = 1 WHERE username = 'lijiguang';

🤔 为什么用 is_admin 而不是 role 表？

你可能想：如果要支持更多角色（如"版主"、"VIP"），是不是应该建一个 roles 表？

对于个人网站，两种角色足够用了。is_admin 字段是最简单的实现——0 或 1，不需要额外的表和 JOIN。

但如果未来需要扩展（如"版主"可以管理特定板块），就需要 roles 表 + user_roles 关联表。这就是从 RBAC0 升级到 RBAC1 的过程。设计时要考虑"演进"——简单方案满足当前需求，同时预留扩展空间。

9.3 管理员中间件

创建一个可复用的管理员检查中间件：

function requireAdmin(req, res, next) {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  db.get(
    'SELECT is_admin FROM users WHERE id = ?',
    [req.session.userId],
    (err, user) => {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      
      if (!user || !user.is_admin) {
        return res.status(403).json({ error: '需要管理员权限' });
      }
      
      next();  // 是管理员，继续执行
    }
  );
}

中间件的设计思路："先验证，后放行"。如果验证失败，直接返回错误响应，不调用 next()；如果验证通过，调用 next() 让请求进入后续处理。

使用方式：

app.get('/api/admin/stats', requireAdmin, (req, res) => {
  // 只有管理员能执行到这里
  db.get('SELECT COUNT(*) as user_count FROM users', [], (err, userCount) => {
    db.get('SELECT COUNT(*) as post_count FROM posts', [], (err2, postCount) => {
      db.get('SELECT COUNT(*) as blog_count FROM blogs', [], (err3, blogCount) => {
        res.json({
          users: userCount.user_count,
          posts: postCount.post_count,
          blogs: blogCount.blog_count
        });
      });
    });
  });
});
💡 中间件链
Express 的路由支持多个中间件：app.get('/path', middleware1, middleware2, handler)。它们按顺序执行，每个中间件可以选择调用 next() 继续，或结束请求。

9.4 管理员 API

9.4.1 获取所有文章（管理员可删任意文章）

app.get('/api/admin/blogs', requireAdmin, (req, res) => {
  db.all(
    `SELECT blogs.*, users.username
     FROM blogs
     JOIN users ON blogs.user_id = users.id
     ORDER BY blogs.created_at DESC
     LIMIT 50`,
    [],
    (err, rows) => {
      if (err) return res.status(500).json({ error: err.message });
      res.json(rows);
    }
  );
});

9.4.2 管理员删除任意文章

app.delete('/api/admin/blogs/:id', requireAdmin, (req, res) => {
  db.run(
    'DELETE FROM blogs WHERE id = ?',
    [req.params.id],
    function(err) {
      if (err) return res.status(500).json({ error: err.message });
      res.json({ message: '删除成功' });
    }
  );
});

注意：管理员删除不检查 user_id——管理员可以删除任何人的内容。这是 RBAC 的核心：角色决定权限边界。

9.6 数据统计查询详解

9.6.1 COUNT 聚合函数

SELECT COUNT(*) as total FROM users;

COUNT(*) 计算所有行数，COUNT(column) 计算该列非 NULL 的行数。对于计数，COUNT(*) 是最快的——数据库不需要检查列值是否为 NULL。

9.6.2 更多统计查询

-- 今日新增用户
SELECT COUNT(*) as today_users 
FROM users 
WHERE DATE(created_at) = DATE('now');

-- 本月文章数
SELECT COUNT(*) as month_blogs 
FROM blogs 
WHERE strftime('%Y-%m', created_at) = strftime('%Y-%m', 'now');

-- 最活跃作者（发文最多）
SELECT user_id, COUNT(*) as blog_count 
FROM blogs 
GROUP BY user_id 
ORDER BY blog_count DESC 
LIMIT 5;

9.7 知识点回顾

📚 本章核心概念回顾

  • RBAC：基于角色的访问控制。用户有角色，角色有权限。
  • RBAC0/1/2：核心、层级、约束三个级别。
  • requireAdmin：可复用的管理员验证中间件。
  • 中间件链：app.get('/path', middleware1, middleware2, handler)。
  • COUNT(*)：聚合函数，计算行数。
  • GROUP BY：按字段分组聚合。
  • strftime：SQLite 日期格式化函数。
  • 信息隐藏：403/404 不区分具体原因，防止信息泄露。

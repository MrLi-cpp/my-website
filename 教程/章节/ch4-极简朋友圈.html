第四章：极简朋友圈——第一个数据库功能

本章目标：实现一个简单的"朋友圈"功能。用户可以发表文字动态，所有访客都能看到发表的动态。首次接触数据库（SQLite）。

学完后你能做什么：刷新页面能看到之前发表的动态，关闭浏览器再打开，动态还在——数据被永久保存了。

4.1 为什么需要数据库？——从"失忆"到"记住"

前三章我们做的网站都是"失忆"的——页面刷新后，所有状态都重置。这是因为数据只存在浏览器的内存里，没有保存到磁盘。

数据库的作用就是"持久化存储"——把数据写入磁盘，即使服务器重启、浏览器关闭，数据也不会丢失。

📖 故事时间：数据库的诞生

1960 年代，计算机主要用于科学计算（比如计算导弹轨迹）。数据存储用磁带或穿孔卡片，没有"数据库"的概念。

1970 年，IBM 的研究员 Edgar F. Codd 发表了一篇论文《A Relational Model of Data for Large Shared Data Banks》，提出了"关系模型"——用二维表格（行和列）组织数据，用 SQL 语言查询。

这篇论文彻底改变了数据管理。在此之前，数据存储用"层次模型"（树形结构）或"网状模型"（图结构），查询极其复杂。Codd 的关系模型让数据操作变得简单直观。

但 IBM 没有重视这个发明——当时 IBM 已经在用层次模型的 IMS 数据库，不愿意推翻重来。1979 年，Larry Ellison 创立了 Oracle 公司，基于 Codd 的论文开发了第一个商用关系数据库。Oracle 后来成为了全球最大的数据库公司之一，Larry Ellison 也成为了亿万富翁。

Codd 本人没有从关系中获利——他一直留在 IBM 做研究。但他的论文被公认为数据库领域最重要的论文之一，他在 1981 年获得了图灵奖。

4.2 SQLite——零配置的轻量级数据库

4.2.1 为什么选择 SQLite？

常见的数据库有：

数据库特点适用场景
SQLite零配置、单文件、嵌入式个人项目、小型应用、原型开发
MySQL开源、成熟、社区大中小型网站、电商
PostgreSQL功能强大、标准兼容企业级应用、复杂查询
MongoDBNoSQL、文档存储、灵活大数据、快速迭代

我们选择 SQLite 的原因是：

  • 零配置：不需要安装数据库服务器，不需要配置用户权限
  • 单文件：整个数据库就是一个 .sqlite 文件，备份只需复制文件
  • 足够用：对于个人网站的流量，SQLite 完全够用
  • 轻量级：node-sqlite3 包体积很小，启动速度快

🤔 SQLite 能支撑多大的流量？

SQLite 的官网上有一句话："SQLite works great as the database engine for most low to medium traffic websites."

SQLite 能处理的并发量取决于操作类型：读操作可以并发，写操作需要加锁。对于个人网站，并发量通常不超过几十个，SQLite 完全没问题。

只有当流量达到每天百万级访问时，你才需要考虑 MySQL 或 PostgreSQL。在那之前，SQLite 是最简单的选择。

很多大型软件也用 SQLite 做内部数据存储：Chrome 浏览器用 SQLite 存历史记录和 Cookie，微信用 SQLite 存聊天记录，iOS 的 Core Data 底层也是 SQLite。

4.2.2 安装 SQLite 驱动

sqlite3 是 Node.js 连接 SQLite 的驱动包。它把 JavaScript 的函数调用翻译成 SQLite 能理解的 C 语言 API。

4.3 创建数据表——设计朋友圈的数据结构

4.3.1 表的设计思路

朋友圈需要存储什么信息？

  • 谁发的？→ 用户名
  • 发了什么？→ 文字内容
  • 什么时候发的？→ 发布时间

于是我们设计一个 posts 表：

字段名类型说明
idINTEGER主键，自增
contentTEXT动态内容
authorTEXT作者名
created_atDATETIME发布时间

4.3.2 初始化数据库

在 server.js 中添加数据库初始化代码：

// 文件位置：my-website/server.js（第四章版本）

const express = require('express');
const sqlite3 = require('sqlite3').verbose();
const app = express();

// ===== 中间件 =====
app.use(express.json());
app.use(express.static('public'));

// ===== 数据库初始化 =====
const db = new sqlite3.Database('./database.sqlite');

db.serialize(() => {
  db.run(`
    CREATE TABLE IF NOT EXISTS posts (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      content TEXT NOT NULL,
      author TEXT DEFAULT '匿名',
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP
    )
  `);
});

// ===== 路由 =====
// 获取所有动态（按时间倒序）
app.get('/api/posts', (req, res) => {
  db.all(
    'SELECT * FROM posts ORDER BY created_at DESC',
    [],
    (err, rows) => {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      res.json(rows);
    }
  );
});

// 发表动态
app.post('/api/posts', (req, res) => {
  const { content, author } = req.body;
  
  if (!content || content.trim() === '') {
    return res.status(400).json({ error: '内容不能为空' });
  }
  
  db.run(
    'INSERT INTO posts (content, author) VALUES (?, ?)',
    [content, author || '匿名'],
    function(err) {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      res.json({
        id: this.lastID,
        content,
        author: author || '匿名',
        created_at: new Date().toISOString()
      });
    }
  );
});

// ===== 启动 =====
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`服务器已启动！访问 http://localhost:${PORT}`);
});

4.3.3 逐行解析

const db = new sqlite3.Database('./database.sqlite');

创建数据库连接。如果 database.sqlite 文件不存在，SQLite 会自动创建它。

db.serialize(() => { ... });

serialize 表示"串行执行"——里面的 SQL 语句按顺序执行，不会并发。这是初始化时的最佳实践，防止 CREATE TABLE 和 INSERT 互相干扰。

CREATE TABLE IF NOT EXISTS

IF NOT EXISTS 表示"如果表不存在就创建，存在就什么都不做"。这样服务器重启时不会报错（表已经存在了）。

id INTEGER PRIMARY KEY AUTOINCREMENT

这是 SQLite 的自增主键语法。

  • INTEGER：整数类型
  • PRIMARY KEY：主键，唯一标识每一行
  • AUTOINCREMENT：插入新行时自动递增

content TEXT NOT NULL

NOT NULL 表示这个字段不能为空。如果插入数据时不提供 content，SQLite 会报错。

author TEXT DEFAULT '匿名'

DEFAULT 表示默认值。如果插入数据时不提供 author，自动设为 '匿名'。

created_at DATETIME DEFAULT CURRENT_TIMESTAMP

DEFAULT CURRENT_TIMESTAMP 表示默认值为当前时间。插入数据时如果不提供时间，自动记录插入时刻。

🤔 为什么用 id 而不是直接用内容做标识？

你可能想：内容本身就是唯一的，为什么还需要 id？

问题在于：内容可能重复（比如两条"今天天气真好"），内容可能很长（不适合做索引），内容可能修改（改内容后标识也变了）。

id 是"代理键"（Surrogate Key）——一个与业务无关的唯一标识。它的优点是：

  • 永远不变（即使内容修改，id 不变）
  • 长度固定（整数比长文本高效得多）
  • 适合关联（其他表引用 posts 时用 id）

几乎所有数据库表都应该有一个自增 id 作为主键。这是数据库设计的最佳实践。

4.4 SQL 查询详解

4.4.1 SELECT * FROM posts

SELECT 是 SQL 中用于查询数据的关键字。

SELECT * FROM posts ORDER BY created_at DESC

逐词解析：

  • SELECT：我要查询数据
  • *：所有字段（id, content, author, created_at）
  • FROM posts：从 posts 表查
  • ORDER BY created_at：按 created_at 字段排序
  • DESC：降序（Descending），最新的在前面

如果不加 ORDER BY，返回的顺序是不确定的（取决于 SQLite 的内部存储顺序）。所以查询时通常都要显式排序。

4.4.2 INSERT INTO posts

INSERT INTO posts (content, author) VALUES (?, ?)

逐词解析：

  • INSERT INTO posts：我要往 posts 表里插入数据
  • (content, author)：要插入的字段
  • VALUES (?, ?)：具体的值，用 ? 作为占位符

? 是"参数占位符"，防止 SQL 注入攻击。我们会在后面展开讲。

4.6 XSS 攻击——为什么需要 escapeHtml？

4.6.1 什么是 XSS？

XSS（Cross-Site Scripting，跨站脚本攻击）是 Web 安全中最常见的攻击方式之一。

攻击原理：如果用户输入的内容包含 HTML 代码，而你直接把它插入到页面中，浏览器会执行这些代码。

比如，用户在朋友圈输入：

<script>alert('你被攻击了！')</script>

如果直接显示在页面上，所有看到这条动态的人都会弹出一个对话框。更严重的攻击可以：

  • 窃取用户的 Cookie（从而冒充用户身份）
  • 跳转到钓鱼网站
  • 在页面上植入恶意代码

📖 故事时间：历史上著名的 XSS 攻击

2005 年，Samy Kamkar 在 MySpace 上发布了一个 XSS 蠕虫。他在个人资料里嵌入了一段 JavaScript 代码，当其他用户访问他的页面时，这段代码会自动：1）把 Samy 加为好友；2）在受害者自己的资料里也嵌入同样的代码。

这个蠕虫在 20 小时内感染了超过 100 万个 MySpace 用户，让 Samy 成为了 MySpace 上最受欢迎的人。MySpace 不得不关闭网站来清理蠕虫。Samy 后来被 FBI 起诉，罚了社区服务和赔偿。

这个故事告诉我们：XSS 不仅仅是"弹个对话框"那么简单，它可以传播、感染、造成大规模破坏。

4.6.2 如何防御 XSS？

最简单的防御方法是"转义"（Escape）——把 HTML 特殊字符转换成安全的形式：

原始字符转义后
<&lt;
>&gt;
&&amp;
"&quot;

转义后，<script> 变成 &lt;script&gt;，浏览器会把它当作纯文本显示，而不会执行。

我们的 escapeHtml 函数利用了一个巧妙的技巧：创建一个 div，把文本设为 textContent（纯文本），然后读取 innerHTML（HTML 表示）。浏览器会自动完成转义。

第三章：主页四联屏——用 CSS 创造沉浸式滑动体验

本章目标：将主页升级为四个全屏区块，用户滚动时每个区块"占满整个屏幕"，形成电影般的画面切换效果。

学完后你能看到什么：滚动鼠标时，主页像电影画面一样一页一页切换，每个页面占满整个屏幕。四个页面分别是：霁光主页、朋友圈入口、博客入口、关于我。

3.1 为什么要做"全屏区块"？

传统的网页是"长页面"——内容从上到下排列，滚动是连续的。全屏区块的设计打破了这种连续性：滚动一下，画面"跳"到下一页，就像翻 PPT 一样。

这种设计在 2010 年后流行起来，尤其是苹果的产品页面（比如 iPhone 介绍页）。苹果发现：全屏画面 + 滚动切换，比长页面的"信息轰炸"更能传达产品的质感和理念。

📖 故事时间：全屏滚动的设计哲学

2010 年，苹果发布了 iPhone 4 的产品页面。设计师们决定采用一种全新的展示方式：每个 iPhone 特性占一个全屏，滚动时画面平滑切换。

这个设计的灵感来自杂志。传统网页是"报纸"——信息密集，需要快速浏览。但高端产品需要"杂志"的体验——大幅图片、留白、沉浸感。苹果把网页从"报纸"变成了"杂志"。

这种设计后来被总结为"全屏滚动"（Fullscreen Scrolling）或"视差滚动"（Parallax Scrolling）。它有两个核心特点：

  1. 分页感：每个屏幕是一个独立的"页"，有自己的主题
  2. 沉浸感：全屏画面让用户专注当前内容，不被其他元素干扰

霁光的四联屏也是这个思路：第一屏是品牌（霁光），第二屏是朋友圈（生活），第三屏是博客（思考），第四屏是关于我（人物）。四个屏幕像四幅画作，构成完整的叙事。

3.2 CSS 变量——让配色方案可维护

在前两章中，我们把颜色直接写在 CSS 里（比如 color: #c9a96e）。但这样做有个问题：如果你想换配色，需要把所有 #c9a96e 都找出来改掉。如果有 50 处用了这个颜色，改起来很痛苦。

CSS 变量（Custom Properties）解决了这个问题。你可以定义一个变量，然后在多处引用它。改颜色时只需要改变量的定义。

3.2.1 定义 CSS 变量

CSS 变量用 -- 前缀定义，用 var() 函数引用：

/* 定义在 :root 选择器中，全局可用 */
:root {
  --bg-primary: #0c0c0e;      /* 主背景 */
  --bg-secondary: #141417;     /* 次背景 */
  --bg-card: rgba(255,255,255,0.02);  /* 卡片背景 */
  
  --accent-gold: #c9a96e;      /* 金色强调 */
  --accent-red: #c0392b;       /* 红色强调 */
  --accent-warm: #a09a90;      /* 暖灰 */
  
  --text-primary: #f0ece6;     /* 主文字 */
  --text-secondary: #a09a90;   /* 次文字 */
  --text-muted: #6a6560;       /* 弱化文字 */
  
  --border: rgba(255,255,255,0.06);  /* 边框色 */
  
  --font-serif: "Noto Serif SC", "PingFang SC", serif;
  --font-sans: -apple-system, BlinkMacSystemFont, "PingFang SC", "Microsoft YaHei", sans-serif;
}

:root 是什么？它选中的是文档的根元素（对于 HTML 来说就是 <html>）。把变量定义在 :root 里意味着所有元素都能访问这些变量。

🤔 为什么用 -- 前缀而不是 $ 或 @？

Sass 和 Less 等 CSS 预处理器用 $variable 或 @variable 表示变量。CSS 原生变量选择 --variable 是为了避免冲突——预处理器不会用 -- 开头，所以两者可以共存。

而且 -- 在命令行中有"选项"的含义（比如 node --version），CSS 工作组觉得这个语义很合适：变量就是"可配置的选项"。

CSS 变量于 2016 年被所有主流浏览器支持，彻底改变了 CSS 的组织方式。今天，没有 CSS 变量的项目会被认为"不够现代"。

3.2.2 使用 CSS 变量

/* 以前 */
.home-title {
  color: #c9a96e;
}

/* 现在 */
.home-title {
  color: var(--accent-gold);
}

/* 带默认值：如果 --accent-gold 不存在，用 #d4af37 */
.home-title {
  color: var(--accent-gold, #d4af37);
}

3.2.3 为什么现在才引入 CSS 变量？

你可能好奇：既然 CSS 变量这么好，为什么前两章没讲？

因为 CSS 变量是一个"进阶"概念。初学者如果一开始就接触变量，可能会混淆"变量名"和"属性值"。先学会直接写颜色值，再引入变量，理解会更深刻。

而且 CSS 变量的最佳使用场景就是"多处复用"——四联屏有四个区块，每个区块可能用到相同的颜色，这时候变量的价值才真正体现出来。

3.3 全屏区块的实现——scroll-snap

实现全屏滚动有两个方案：

  • JavaScript 方案：监听 scroll 事件，计算当前位置，强制滚动到最近的整屏。缺点：性能差，滚动不流畅。
  • CSS 方案（scroll-snap）：浏览器原生支持，性能最好，代码最少。优点：硬件加速，平滑自然。

我们选择 CSS 方案。

📖 故事时间：scroll-snap 的坎坷标准化之路

2015 年，苹果在 iOS Safari 中率先实现了 CSS Scroll Snapping（当时叫 -webkit-scroll-snap）。这是一个"私有前缀"特性，只有 Safari 能用。

其他浏览器厂商看到了这个特性的价值，但想要一个"标准化的"版本。W3C CSS 工作组花了两年时间讨论 API 设计——比如：snap 应该发生在元素还是容器层面？怎么控制 snap 的"严格程度"？

2017 年，CSS Scroll Snap Module Level 1 正式发布。2019 年，所有主流浏览器都支持了标准语法。这个特性从苹果私有的实验性功能，变成了 Web 标准的一部分——这是"浏览器前缀→标准化"的典型案例。

这也是为什么我们写代码时不需要加 -webkit- 前缀了。但在 2016 年，你还需要写两套代码（一套带前缀，一套标准语法）来兼容所有浏览器。

3.3.1 核心 CSS 属性

/* 容器设置 */
.scroll-container {
  height: 100vh;                    /* 容器高度 = 视口高度 */
  overflow-y: scroll;               /* 允许垂直滚动 */
  scroll-snap-type: y mandatory;    /* Y轴方向，强制 snap */
}

/* 每个区块设置 */
.screen {
  height: 100vh;                    /* 每个区块 = 一屏高度 */
  scroll-snap-align: start;         /* 滚动时对齐到区块的起始位置 */
}

逐行解析：

scroll-snap-type: y mandatory：告诉浏览器，这个容器的滚动需要"snap"（吸附）。y 表示只在垂直方向吸附。mandatory 表示"强制"——滚动结束后必须停在一个 snap 点上，不能停在两个区块之间。

另一个可选值是 proximity（接近），表示"如果滚动停止位置接近 snap 点，就吸附过去；否则保持原样"。mandatory 更适合全屏滚动，因为它保证了"一页一页"的体验。

scroll-snap-align: start：每个区块定义自己的"snap 对齐方式"。start 表示区块的顶部对齐到容器的顶部。你也可以用 end（底部对齐）或 center（居中对齐）。

🤔 为什么 scroll-snap 比 JavaScript 方案好？

JavaScript 监听 scroll 事件的问题是：scroll 事件触发频率极高（每 16ms 一次，对应 60fps）。如果在事件处理函数里做复杂的计算，会导致"卡顿"（jank）——页面掉帧。

scroll-snap 由浏览器底层实现，走的是硬件加速路径（GPU 合成层）。浏览器在渲染线程之外计算 snap 位置，不影响主线程的 JavaScript 执行。这就是为什么 CSS 动画通常比 JavaScript 动画更流畅。

这也是现代 Web 开发的一个重要原则：能用 CSS 做的，不要用 JavaScript。CSS 动画和过渡由浏览器优化过，性能更好；JavaScript 动画需要自己管理帧率、缓动曲线、硬件加速，容易出问题。

3.5 动画——让页面"活"起来

3.5.1 @keyframes 和 animation

我们给"向下滚动"提示加了一个弹跳动画：

@keyframes bounce {
  0%, 100% { transform: translateY(0); }
  50% { transform: translateY(10px); }
}

@keyframes 定义了一个动画序列。0% 是动画开始时的状态，50% 是中间状态，100% 是结束状态。浏览器会自动计算中间帧，形成平滑动画。

应用动画：

.scroll-hint {
  animation: bounce 2s infinite;
  /*            名称   时长  循环次数 */
}

infinite 表示无限循环。你也可以写 3 表示只播放 3 次。

📖 故事时间：CSS 动画的历史

CSS 动画的发展经历了三个阶段：

第一阶段（1990s）：GIF 动画。 早期的网页动画就是 GIF 图片——一个文件包含多帧图像，循环播放。缺点是画质差、不能交互、文件大。

第二阶段（2000s）：Flash。 Adobe Flash 插件提供了强大的矢量动画和交互能力，一度占据了 Web 动画的统治地位。2005 年 YouTube 就用 Flash 播放视频。但 Flash 的问题是：需要安装插件、安全性差（漏洞无数）、移动设备不支持。乔布斯 2010 年发布的《Thoughts on Flash》公开信宣判了 Flash 的死刑。

第三阶段（2010s+）：CSS3 + JavaScript。 2011 年，CSS3 Animations 和 Transitions 被主流浏览器支持。配合 HTML5 Canvas 和 WebGL，浏览器原生就能实现复杂的动画效果，不再需要插件。

今天我们用 @keyframes 做简单动画，用 CSS Transitions 做状态变化，用 JavaScript 的 requestAnimationFrame 做复杂交互。Flash 已经完全退出了历史舞台，连 Adobe 都在 2020 年停止了对 Flash 的支持。

3.5.2 按钮 hover 效果

.screen-btn {
  background: transparent;
  border: 1px solid var(--accent-gold);
  color: var(--accent-gold);
  transition: all 0.3s ease;
}

.screen-btn:hover {
  background: var(--accent-gold);
  color: var(--bg-primary);
}

这个 hover 效果的设计思路是"反色"：默认状态下按钮是透明的，文字是金色；hover 时背景变成金色，文字变成深色。这种"填充式"的反馈非常直观。

🤔 为什么选择"反色"而不是"发光"？

"发光"效果（box-shadow + border-color 变化）适合卡片这种"大面积"元素。但对于按钮这种"小面积"元素，发光效果不够明显——金色的光晕在深色背景上几乎看不见。

"反色"（颜色互换）是按钮 hover 的经典方案。它的原理是"对比"——hover 前后颜色差异越大，用户的感知越强。金色(#c9a96e) 和深色(#0c0c0e) 的对比度足够高，hover 效果一目了然。

这体现了 UI 设计的一个原则：根据元素大小选择合适的反馈方式。小元素用颜色变化，大元素用阴影/缩放，整个页面用过渡动画。

3.6 为什么 body 要 overflow: hidden？

你可能注意到我们给 body 加了 overflow: hidden，同时又给 scroll-container 加了 overflow-y: scroll。为什么不直接在 body 上滚动？

原因是：scroll-snap 需要作用在"滚动容器"上。如果让 body 滚动，scroll-snap 会作用在整个页面上——但页面上可能有其他元素（比如弹窗、导航栏）也会参与滚动，导致 snap 行为混乱。

通过把滚动限制在一个明确的容器（.scroll-container）里，我们实现了"隔离"——只有这个容器里的子元素会 snap，外部元素不受影响。

这也方便我们以后添加"非全屏"的内容。比如页面底部加一个 footer，它不在 .scroll-container 里，所以不受 snap 影响。

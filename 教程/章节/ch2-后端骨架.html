第二章：后端骨架——让服务器"听懂"浏览器的请求

本章目标：理解后端是什么，用 Express 搭建可处理 API 请求的服务器骨架，学会路由、中间件、JSON 响应。

学完后你能做什么：浏览器能向服务器"要数据"，服务器能"给数据"，双方用 JSON 格式对话。

2.1 后端是什么？——餐厅服务员的比喻

想象你走进一家餐厅：

  • 你坐在座位上，看菜单（网页前端）
  • 你叫服务员过来，说"我要一份牛排"（发送请求）
  • 服务员把单子递给厨房（后端处理）
  • 厨房做菜（数据库查询、业务逻辑）
  • 服务员把菜端给你（返回响应）

在这个比喻里：

餐厅网站
菜单（菜品图片、价格、描述）前端（HTML/CSS/JS 构成的界面）
服务员后端服务器（Express）
厨房数据库 + 业务逻辑
食材数据（用户信息、文章内容）

前端负责"展示"，后端负责"处理"。前端是脸面，后端是大脑。

📖 故事时间：为什么网站需要后端？

1990 年代的网站几乎都是"静态网站"——HTML 文件放在服务器上，浏览器请求什么就返回什么。这种网站只能展示固定内容，无法交互。

1993 年，NCSA 开发了 CGI（Common Gateway Interface，通用网关接口）。CGI 允许 Web 服务器调用外部程序来处理请求。比如，用户提交一个表单，CGI 调用一个 Perl 脚本，脚本处理数据后生成新的 HTML 返回。

CGI 的问题是：每次请求都要启动一个新的进程。这就像餐厅每来一位客人就新招一个服务员——开销巨大。

1997 年，PHP 出现了。PHP 的解决方案是"把程序嵌在 HTML 里"——服务器常驻内存，请求来了直接处理。这大大提升了性能。

2009 年 Node.js 出现后，JavaScript 也能做后端了。Express 作为 Node.js 最流行的框架，简化了路由、中间件、模板渲染等常见任务。今天我们用 Express，就是在站在 30 年 Web 后端发展的肩膀上。

🤔 为什么前后端要分离？

早期的 PHP/ASP/JSP 时代，前后端是不分离的。同一个文件里既有 HTML 标签，又有数据库查询代码，还有业务逻辑。这种"意大利面条代码"维护起来极其困难。

2005 年后，随着 AJAX 技术成熟，前后端分离成为主流。前端用 JavaScript 调用后端 API 获取数据，然后动态渲染页面。后端只负责数据，前端只负责展示。

好处是什么？

  • 职责清晰：前端工程师专注交互和视觉效果，后端工程师专注数据和性能
  • 并行开发：前后端可以同时开发，通过 API 文档约定接口
  • 技术独立：前端可以用 React/Vue，后端可以用 Python/Java/Node.js，互不影响
  • 多端适配：同一个后端 API 可以同时服务网页、手机 App、小程序

我们的项目也遵循这个原则：public/ 文件夹里是前端代码，server.js 是后端代码，双方通过 JSON API 通信。

2.2 路由——服务器的"菜单"

2.2.1 什么是路由？

路由（Routing）就是"根据 URL 决定执行什么代码"。

想象餐厅的服务员听到不同点菜指令：

  • "我要牛排" → 传递给厨房 A（牛排师傅）
  • "我要寿司" → 传递给厨房 B（日料师傅）
  • "我要账单" → 传递给收银员

在 Express 中：

app.get('/steak', (req, res) => {
  res.send('牛排来了！');
});

app.get('/sushi', (req, res) => {
  res.send('寿司来了！');
});

app.get('/bill', (req, res) => {
  res.send('账单来了！');
});

app.get('/steak', ...) 表示：当浏览器用 GET 方法访问 /steak 路径时，执行后面的函数。

2.2.2 HTTP 方法——GET vs POST

HTTP 定义了多种"方法"（也叫动词），表示请求的目的：

方法含义用途是否安全*
GET获取浏览页面、查询数据是
POST创建提交表单、注册账号否
PUT更新修改资料、编辑文章否
DELETE删除删除文章、注销账号否
PATCH部分更新修改部分字段否

*"安全"指的是：多次执行不会产生副作用。GET 请求不会修改服务器数据，POST/PUT/DELETE 会。

🤔 为什么 GET 和 POST 要分开？

这个区分来自 RESTful 架构的设计理念。REST（Representational State Transfer）是 Roy Fielding 在 2000 年博士论文中提出的架构风格，主张用 HTTP 方法表达操作意图。

REST 的核心思想：URL 是资源（名词），HTTP 方法是操作（动词）。比如：

  • GET /users/1 → 获取 ID 为 1 的用户
  • POST /users → 创建一个新用户
  • PUT /users/1 → 修改 ID 为 1 的用户
  • DELETE /users/1 → 删除 ID 为 1 的用户

这种设计的好处是语义清晰。看 API 就能知道它在做什么，不需要读文档。而且浏览器、CDN、搜索引擎都能利用 GET 的"安全"特性做缓存。

我们的项目虽然没有严格遵循 RESTful（比如有些操作直接用 POST），但基本理念是一致的。

2.3 中间件——Express 的灵魂

2.3.1 什么是中间件？

中间件（Middleware）是 Express 最强大的设计。它是一个函数，接收请求对象（req）、响应对象（res）、以及一个next() 函数。

中间件可以做三件事：

  • 处理请求（比如解析 JSON 数据）
  • 拒绝请求（比如权限检查不通过）
  • 放行给下一个中间件（调用 next()）
// 自定义中间件示例
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next(); // 放行，继续执行后续的中间件或路由
});

这个中间件的作用是：记录每个请求的到达时间、HTTP 方法和 URL。调用 next() 表示"我处理完了，继续下一个"。如果不调用 next()，请求就会"卡住"，浏览器永远等不到响应。

📖 故事时间：中间件模式的起源

中间件的概念不是 Express 发明的。它来自软件架构中的"管道-过滤器"模式（Pipes and Filters Pattern）。

这个模式最早出现在 Unix 操作系统的设计中。Unix 的哲学是：每个程序只做一件事，然后把多个程序用"管道"连接起来。

比如：

cat file.txt | grep "error" | sort | uniq -c

这条命令的意思是：读取文件 → 过滤包含 "error" 的行 → 排序 → 统计重复次数。每个程序只做一件事，通过管道传递数据。

Express 的中间件就是这个思路：每个中间件处理请求的一个方面（日志、解析、认证），通过 next() 传递控制权。

这种设计的好处是可组合。你可以像搭积木一样组合中间件：日志中间件 + 认证中间件 + 解析中间件 + 路由处理。想加功能就加一个中间件，想删功能就删掉它，不影响其他部分。

2.3.2 常用内置中间件

Express 提供了几个常用内置中间件：

中间件作用使用方式
express.json()解析请求体中的 JSON 数据app.use(express.json())
express.urlencoded()解析表单提交的 x-www-form-urlencoded 数据app.use(express.urlencoded({ extended: true }))
express.static()托管静态文件（CSS/JS/图片）app.use(express.static('public'))

为什么需要 express.json()？

当浏览器发送 POST 请求时，请求体里可能带着 JSON 数据。比如注册账号时，浏览器发送：

POST /api/register
Content-Type: application/json

{
  "username": "jiguang",
  "password": "secret123"
}

如果不加 express.json()，服务器收到的 req.body 是 undefined，完全读不到用户名和密码。这个中间件会自动把请求体中的 JSON 字符串解析成 JavaScript 对象。

🤔 为什么默认不自动解析 JSON？

你可能会想：既然解析 JSON 这么常用，为什么 Express 不默认开启？

这是 Express "极简主义"哲学的体现。Express 的核心非常轻量，所有功能都是可选的。这样做有几个好处：

  • 性能：不需要的功能不加载，减少内存占用
  • 灵活：你可以用自己习惯的 JSON 解析库（比如 body-parser 的替代品）
  • 安全：JSON 解析有潜在的安全风险（比如大体积 JSON 导致内存耗尽），默认关闭意味着你需要显式决定"我要这个风险"

这种"显式优于隐式"（Explicit is better than implicit）的设计哲学来自 Python 的 Zen of Python，也是很多优秀框架的共同选择。

2.4 JSON 响应——前后端的"通用语言"

2.4.1 什么是 JSON？

JSON（JavaScript Object Notation，JavaScript 对象表示法）是一种轻量级的数据交换格式。它的语法和 JavaScript 对象几乎一样：

{
  "name": "霁光",
  "age": 20,
  "isAdmin": false,
  "hobbies": ["编程", "写作", "摄影"]
}

JSON 的规则很简单：

  • 键必须是字符串（用双引号包裹）
  • 值可以是字符串、数字、布尔值、null、数组、对象
  • 不支持函数、日期、undefined（JSON 没有这些类型）

📖 故事时间：JSON 的诞生

2001 年，Douglas Crockford（JavaScript 领域的传奇人物）在一家公司做顾问。当时公司内部需要一种数据交换格式，XML 太臃肿，YAML 太复杂。Crockford 说："JavaScript 的对象语法已经很完美了，我们只需要把它标准化。"

他花了一下午定义了 JSON 的语法规范，然后在 2002 年在 json.org 发布。JSON 的"卖点"是极其简单：

  • 人可读：不像二进制格式需要专用工具查看
  • 机器可解析：几乎所有编程语言都有 JSON 解析库
  • 体积小：比 XML 少了很多冗余标签

JSON 迅速取代 XML 成为 Web API 的事实标准。今天，你很难找到一个不支持 JSON 的 Web 服务。Douglas Crockford 因为 JSON 获得了巨大的声誉，但他从未为 JSON 申请专利——他说："JSON 是送给互联网的礼物。"

2.4.2 服务器返回 JSON

app.get('/api/user', (req, res) => {
  res.json({
    name: '霁光',
    bio: '记录生活 · 沉淀思考',
    website: 'http://localhost:3000'
  });
});

res.json() 会自动：

  • 把 JavaScript 对象转换成 JSON 字符串
  • 设置 Content-Type 为 application/json
  • 发送响应

如果不用 res.json()，你需要手动写：

app.get('/api/user', (req, res) => {
  res.setHeader('Content-Type', 'application/json');
  res.send(JSON.stringify({ name: '霁光', bio: '...' }));
});

res.json() 只是帮你封装了这些操作，让你写更少的代码。

2.5 本章实操——搭建后端骨架

步骤1：安装依赖
cd ~/Desktop/my-website
npm install express
步骤2：创建 server.js 骨架
// 文件位置：my-website/server.js

const express = require('express');
const app = express();

// ===== 中间件 =====
app.use(express.json());                    // 解析 JSON 请求体
app.use(express.static('public'));          // 托管静态文件

// ===== 路由 =====
// 测试 API：返回当前时间
app.get('/api/time', (req, res) => {
  res.json({
    time: new Date().toLocaleString('zh-CN'),
    message: '服务器运行正常'
  });
});

// 测试 API：接收 POST 请求
app.post('/api/echo', (req, res) => {
  res.json({
    received: req.body,
    message: '服务器收到了你的数据！'
  });
});

// ===== 启动 =====
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`服务器已启动！访问 http://localhost:${PORT}`);
});

逐行解析

const PORT = process.env.PORT || 3000;

这一行让服务器端口可配置。如果环境变量 PORT 存在（比如部署到服务器时设为 80），就用环境变量的值；否则默认用 3000。

process.env 是 Node.js 提供的环境变量对象。环境变量是操作系统级别的键值对配置，不会写入代码，因此适合放敏感信息（比如数据库密码）。

🤔 为什么用 || 而不是 if-else？

process.env.PORT || 3000 是 JavaScript 的"短路求值"特性。逻辑或运算符 || 的规则是：如果左边是"真值"（truthy），返回左边；否则返回右边。

JavaScript 中，以下值是"假值"（falsy）：

  • false
  • 0
  • ""（空字符串）
  • null
  • undefined
  • NaN

其他所有值都是真值（包括 "0"、[]、{}）。所以当 process.env.PORT 未定义时（undefined），|| 返回 3000。

这种写法比 if (process.env.PORT) { PORT = process.env.PORT; } else { PORT = 3000; } 简洁得多，是 JavaScript 程序员的惯用写法。

步骤3：测试 GET API
启动服务器后，在浏览器访问 http://localhost:3000/api/time，你应该看到： 
{
  "time": "2026/5/15 14:30:00",
  "message": "服务器运行正常"
}
步骤4：测试 POST API
POST 请求不能直接用浏览器地址栏测试（浏览器地址栏只能发 GET 请求）。打开 Chrome DevTools（F12），切换到 Console 标签，输入： 
fetch('/api/echo', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: '霁光', hobby: '编程' })
})
.then(res => res.json())
.then(data => console.log(data));

你应该看到返回：

{
  "received": { "name": "霁光", "hobby": "编程" },
  "message": "服务器收到了你的数据！"
}

2.6 fetch API——前端如何调用后端

fetch 是现代浏览器内置的 HTTP 请求 API。它是 XMLHttpRequest 的替代品，API 更简洁，基于 Promise。

📖 故事时间：从 XMLHttpRequest 到 fetch

1999 年，微软在 IE5 中引入了 XMLHttpRequest（XHR），让浏览器可以在不刷新页面的情况下向服务器请求数据。这是 AJAX（Asynchronous JavaScript and XML）技术的基础。

但 XHR 的 API 设计很糟糕。一个简单的 GET 请求需要写这么多代码：

var xhr = new XMLHttpRequest();
xhr.open('GET', '/api/data', true);
xhr.onreadystatechange = function() {
  if (xhr.readyState === 4 && xhr.status === 200) {
    console.log(xhr.responseText);
  }
};
xhr.send();

readyState 的 4 个状态、status 码判断、回调函数嵌套……代码非常冗长。

2015 年，fetch API 作为 Promise 风格的替代品被标准化。同样的功能只需：

fetch('/api/data')
  .then(res => res.json())
  .then(data => console.log(data));

简洁、链式、符合现代 JavaScript 习惯。今天，除非需要进度条上传等高级功能，fetch 已经完全替代了 XHR。

fetch 的基本用法：

// GET 请求（获取数据）
fetch('/api/time')
  .then(response => response.json())   // 解析 JSON
  .then(data => console.log(data))     // 使用数据
  .catch(error => console.error(error)); // 处理错误

// POST 请求（发送数据）
fetch('/api/echo', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({ name: '霁光' })
});

关键概念：

  • fetch() 返回一个 Promise，表示"未来的结果"
  • .then() 在 Promise 成功时执行
  • .catch() 在 Promise 失败时执行（比如网络断开）
  • response.json() 也是 Promise，表示"解析 JSON 完成后的结果"

🤔 为什么 fetch 的 API 设计成 Promise？

Promise 是 JavaScript 异步编程的核心概念。传统回调函数的问题是"回调地狱"（Callback Hell）——嵌套层级太深，代码难以阅读。

// 回调地狱示例
getData(function(a) {
  getMoreData(a, function(b) {
    getMoreData(b, function(c) {
      getMoreData(c, function(d) {
        console.log(d);
      });
    });
  });
});

Promise 的链式调用让异步代码看起来像同步代码：

getData()
  .then(a => getMoreData(a))
  .then(b => getMoreData(b))
  .then(c => getMoreData(c))
  .then(d => console.log(d));

这种"扁平化"的代码结构是现代 JavaScript 的标配。async/await（ES2017）进一步让异步代码看起来像同步代码，我们在后续章节会用到。

2.7 错误处理——让程序"优雅地失败"

任何程序都会出错。网络断开、数据库连接失败、代码写错……一个好的后端不会让错误信息直接暴露给用户，而是"优雅地处理"。

2.7.1 404 处理

// 放在所有路由的最后
app.use((req, res) => {
  res.status(404).json({
    error: '接口不存在',
    path: req.path
  });
});

这个中间件没有路径限制（app.use 不带路径），它会捕获所有没有被前面路由处理的请求。返回 404 状态码和友好的错误信息。

🤔 为什么 404 中间件要放在最后？

Express 的路由匹配是顺序的。请求从上到下依次匹配，一旦匹配成功就停止。如果 404 中间件放在前面，它会拦截所有请求，后面的路由永远不会执行。

这就像餐厅的服务流程：迎宾 → 点餐 → 上菜 → 送客。你不能把"送客"放在"点餐"前面。

所以 Express 中间件的顺序非常关键。常见的顺序是：

  1. 日志中间件（最先，记录所有请求）
  2. 解析中间件（json、urlencoded）
  3. 静态文件中间件（express.static）
  4. 认证中间件（如果需要）
  5. 业务路由（GET / POST / PUT / DELETE）
  6. 404 处理（最后）

2.7.2 全局错误捕获

// 捕获同步和异步错误
app.use((err, req, res, next) => {
  console.error('服务器错误:', err);
  res.status(500).json({
    error: '服务器内部错误',
    message: process.env.NODE_ENV === 'development' ? err.message : '请稍后重试'
  });
});

这个中间件有四个参数（err, req, res, next），Express 通过参数数量识别这是"错误处理中间件"。它捕获所有抛出的异常，防止服务器崩溃。

process.env.NODE_ENV 的判断是为了安全：开发环境显示详细错误信息方便调试，生产环境只显示"请稍后重试"，防止泄露敏感信息。

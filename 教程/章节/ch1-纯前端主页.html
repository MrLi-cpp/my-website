第一章：纯前端主页——从零开始画一个好看的页面

本章目标：只写 HTML + CSS，不碰后端。完成一个大气的主页，有标题、个人介绍、三个入口卡片（朋友圈/关于我/博客）。

学完后你能看到什么：打开 localhost:3000，深色背景上金色大字"霁光"居中，下面三个白色半透明卡片排成一排。鼠标放上去卡片会平滑放大，边框泛出金色光晕。

1.1 为什么先学前端？

很多编程教程一上来就让你配数据库、写 API、搭服务器……结果折腾半天，屏幕上除了错误提示什么都没有。这对于初学者来说非常打击积极性——你花了三小时，只学会怎么安装软件，连一个"看得见"的东西都没做出来。

我们的思路是反过来的：先做出看得见的东西，再学背后的原理。就像学画画，老师不会让你先学颜料化学成分，而是直接给你画笔和纸，让你画一朵花。画完之后，你自然会对"为什么这个颜色混那个颜色变成紫色"产生好奇——这时候再讲原理，你听得进去。

📖 故事时间：HTML 和 CSS 是怎么诞生的？

1989年，Tim Berners-Lee 发明万维网时，网页只有纯文本。你可以用 <h1> 标签标记标题，用 <p> 标记段落，但页面的样式完全由浏览器决定。比如标题多大、什么颜色、段落的行间距——用户改不了，浏览器厂商说了算。

1993年，NCSA Mosaic 浏览器（第一个流行的图形界面浏览器）的团队想要一种方式，让用户控制网页的样式。一位叫 Håkon Wium Lie 的挪威程序员在 CERN 工作期间，提交了一份提议：Cascading HTML Style Sheets（层叠 HTML 样式表）。

"层叠"（Cascading）是什么意思？想象一个公司的组织架构：总公司有统一的 dress code（正装），但某个分公司允许穿休闲装，某个部门的经理又允许周五穿拖鞋。最终每个员工的着装，是"总公司→分公司→部门→个人"层层叠加的结果。

CSS 的层叠机制也是这个逻辑：浏览器有默认样式，用户可以在浏览器设置里覆盖，网页作者可以用 style 标签覆盖，还可以用 !important 强行覆盖。优先级从高到低：

  1. !important 声明（最高）
  2. 内联样式（style="..."）
  3. ID 选择器（#xxx）
  4. 类/属性/伪类选择器（.xxx）
  5. 标签/伪元素选择器（div）
  6. 浏览器默认样式（最低）

Håkon 的提议最初遭到了 Tim Berners-Lee 的反对。Tim 认为样式应该通过 HTML 属性来控制（比如 <font color="red">），而不是引入一套独立的语言。但浏览器厂商们（特别是 Netscape 和 Microsoft）发现，独立的样式语言更有利于统一标准。1996年，CSS Level 1 正式发布。

今天，Håkon Wium Lie 是 CSS 标准的共同编辑之一，他在 Opera 浏览器公司工作了十多年，推动了 CSS 的现代化。CSS 从最初只能改颜色和字体，发展到今天能做动画、3D 变换、Grid 布局——但"层叠"的核心理念始终没变。

1.2 HTML 骨架——页面的"骨骼"

1.2.1 什么是 HTML？

HTML（HyperText Markup Language，超文本标记语言）不是编程语言，而是标记语言。它的作用是给文本"打标签"，告诉浏览器"这是标题""这是段落""这是链接"。

"超文本"是什么意思？普通的文本是线性的——你一行一行往下读。超文本可以跳转——点击一个词就能跳到另一个文档。这就是 <a>（anchor，锚点）标签的由来。

🤔 为什么叫"标记"语言？

想象你在写一篇论文，用荧光笔给不同类型的内容做标记：黄色标记重要结论，蓝色标记引用文献，绿色标记待办事项。HTML 标签就是"荧光笔"——<h1> 标记"这是大标题"，<p> 标记"这是普通段落"。

浏览器看到标记后，会按照约定好的方式显示内容。<h1> 默认显示为 2em 大小的粗体字，<strong> 默认显示为粗体。

但你也可以用 CSS 覆盖这些默认显示方式——这就是为什么 HTML 和 CSS 是分开的。HTML 负责"这是什么"，CSS 负责"长什么样"。这种分离让同一份内容可以适配不同的设备（电脑、手机、打印机）。

1.2.2 DOCTYPE 和文档结构

每个 HTML 文件的第一行都应该是：

<!DOCTYPE html>

这行代码告诉浏览器："这是一个 HTML5 文档，请用标准模式渲染。"

📖 故事时间：DOCTYPE 的奇怪语法

<!DOCTYPE html> 的语法看起来有点怪——为什么不写成 <html version="5"> 之类更直观的格式？

答案要追溯到 1997 年的 HTML 4.01。那时候的 DOCTYPE 长这样：

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">

这个语法来自 SGML（标准通用标记语言），而 SGML 又来自更早的 GML（通用标记语言）。GML 是 IBM 在 1960 年代发明的，它的 DOCTYPE 声明就采用了 <!... 的格式。

2008 年，HTML5 的制定者们想简化 DOCTYPE。他们本可以彻底废弃这个怪异的语法，但考虑到：

  • 老旧的浏览器（比如 IE6）看到不认识的 DOCTYPE 会进入"怪异模式"（Quirks Mode），用 1990 年代的 bug 渲染页面
  • <!DOCTYPE html> 虽然语法怪异，但所有浏览器都认识

于是他们保留了 DOCTYPE，但把它缩短到最短。这就是为什么我们今天还在写 <!DOCTYPE html>——它本质上是一个"触发标准模式的开关"，没有其他实际意义。

1.2.3 lang 和 charset

<html lang="zh-CN"><head>
  <meta charset="UTF-8">

lang="zh-CN" 指定页面语言是简体中文。这有什么实际作用？

  • 屏幕阅读器（给视障人士用的读屏软件）会根据 lang 决定用什么语音朗读
  • 浏览器的翻译功能会根据 lang 决定是否提示"翻译此页"
  • 搜索引擎会根据 lang 决定把页面展示给哪个语言区的用户

charset="UTF-8" 指定字符编码为 UTF-8。UTF-8 是目前最通用的字符编码，它用 1-4 个字节表示一个字符。ASCII 字符（英文字母、数字）用 1 个字节，中文用 3 个字节，emoji 用 4 个字节。

🤔 为什么 UTF-8 成为了标准？

在 UTF-8 之前，中文网页常用 GB2312 或 GBK 编码。这些编码只支持中文和部分符号，如果要显示日文、韩文、阿拉伯文，就得用完全不同的编码。

更糟糕的是：如果浏览器用错误的编码解析页面，中文就会变成乱码（俗称"锟斤拷"——这是 Unicode 替换字符 U+FFFD 在 GBK 编码下的显示结果）。

UTF-8 解决了所有问题：一个编码覆盖全世界所有语言，而且是 ASCII 的超集（纯英文的 UTF-8 文件和 ASCII 文件完全一样）。2007年，UTF-8 成为 Web 上使用最多的编码。今天，超过 95% 的网页使用 UTF-8。

这就是为什么你绝对不能忘记写 <meta charset="UTF-8">——没有它，中文就有概率变成乱码。

1.2.4 viewport——移动端适配的关键

<meta name="viewport" content="width=device-width, initial-scale=1.0">

这行代码是移动互联网时代的产物。

📖 故事时间：为什么 iPhone 刚出来时网页都很小？

2007 年，第一代 iPhone 发布。它的屏幕宽度是 320 像素。但当时的大多数网页是为 1024 像素宽的桌面显示器设计的。

苹果的解决方案很巧妙：Safari 浏览器假装自己的宽度是 980 像素（桌面显示器的常见宽度），把整个网页缩小到 320 像素的屏幕上显示。这样网页布局不会乱，但文字变得非常小，需要双指放大才能阅读。

2010 年，苹果在 iOS Safari 中引入了 viewport meta 标签。开发者可以告诉浏览器："不要用 980 像素来渲染我，用设备的真实宽度。"

width=device-width 就是"宽度 = 设备宽度"。initial-scale=1.0 是"初始缩放比例 = 100%"，防止某些浏览器自动缩放。

这个设计后来被所有移动浏览器采纳，成为了响应式网页的基础。没有 viewport，你的网页在手机上永远是被缩小的桌面版。

1.3 CSS 盒子模型——理解页面布局的基石

1.3.1 什么是盒子模型？

CSS 把每个 HTML 元素都视为一个"盒子"。这个盒子由四层组成：

┌─────────────────────────────┐
│         margin（外边距）       │  ← 盒子与其他盒子的间距
│   ┌─────────────────────┐   │
│   │      border（边框）  │   │  ← 盒子的边框
│   │   ┌─────────────┐   │   │
│   │   │ padding（内边距）│   │   │  ← 边框与内容的间距
│   │   │   ┌─────┐   │   │   │
│   │   │   │content│   │   │   │  ← 实际内容（文字、图片等）
│   │   │   └─────┘   │   │   │
│   │   └─────────────┘   │   │
│   └─────────────────────┘   │
└─────────────────────────────┘

这四层的关系：

层作用类比
content实际内容（文字、图片）房间的家具
padding内容与边框的距离家具与墙壁的距离
border边框墙壁
margin盒子与其他盒子的距离房间与隔壁房间的距离

🤔 为什么要分成四层？

盒子模型的设计体现了"关注点分离"的理念。margin 负责"与邻居的关系"，padding 负责"内部空间"，border 负责"边界可视化"，content 负责"实际内容"。

这种分离让你可以独立控制每个方面：想让卡片内部更宽敞？改 padding。想让两个卡片离得更远？改 margin。想让边框更明显？改 border。想换内容？改 content。

如果没有这个分层，你就只能用 margin 做所有事——那 padding 和 border 的效果就实现不了。早期的 HTML 表格布局（用 <table> 做页面布局）就有这个问题：间距控制极其笨拙，设计师不得不用各种 hack（比如透明 spacer GIF 图片）来模拟 margin 和 padding。

CSS 盒子模型的出现，让网页设计从"拼积木"变成了"搭建筑"。

1.3.2 为什么 width/height 默认不包含 padding 和 border？

默认情况下（box-sizing: content-box），你设置的 width 和 height 只包含 content 区域。这意味着：

.entry-card {
  width: 140px;           /* 这只是 content 的宽度！ */
  padding: 20px;          /* 左右各加 20px */
  border: 1px solid red;  /* 左右各加 1px */
  /* 实际占用宽度 = 140 + 20*2 + 1*2 = 182px */
}

这种设计被几乎所有设计师认为是反直觉的。你明明说了 width=140，结果它占了 182？

解决方案就是我们在第零章用过的 box-sizing: border-box：

* {
  box-sizing: border-box;
}

.entry-card {
  width: 140px;           /* 现在这是总宽度！ */
  padding: 20px;
  border: 1px solid red;
  /* content 宽度 = 140 - 20*2 - 1*2 = 98px */
}

border-box 让 width/height 表示"整个盒子占用的空间"，padding 和 border 从内部扣除。这样 width: 140px 就真的占用 140px，不会有意外。

1.4 Flexbox 布局——现代网页的一维布局方案

1.4.1 为什么需要 Flexbox？

在 Flexbox 出现之前，网页布局主要靠三种方式：

  • 表格布局（1990年代）：用 <table> 标签做布局。缺点是语义错误（表格应该放表格数据），而且极其不灵活。
  • 浮动布局（2000年代）：用 float: left 让元素并排。缺点是需要"清除浮动"hack，而且无法垂直居中。
  • 定位布局（2000年代）：用 position: absolute 精确定位。缺点是元素脱离文档流，布局极易破碎。

这三种方式都有一个共同问题：垂直居中极其困难。

在浮动布局时代，实现垂直居中需要知道父元素的高度，然后用 margin-top: calc(50% - 高度/2) 之类的计算——如果高度变化，CSS 就要重写。

📖 故事时间：Flexbox 的漫长诞生

2009年，W3C 发布了 Flexbox 的第一个工作草案。但第一个实现（WebKit 浏览器引擎）用的语法和现在完全不同：

display: box; box-orient: horizontal;

2011年，语法大改：

display: flexbox; flex-pack: center;

2012年，再次大改，终于稳定为现在的语法：

display: flex; justify-content: center; align-items: center;

这个反复修改的过程让开发者很头疼——要写三套 CSS 前缀才能兼容所有浏览器。这就是著名的"浏览器前缀地狱"：

display: -webkit-box; display: -moz-box; display: -ms-flexbox; display: -webkit-flex; display: flex;

幸运的是，今天你只需要写 display: flex; 就够了。所有现代浏览器（Chrome、Firefox、Safari、Edge）都支持标准语法。

Flexbox 的名字来源于"Flexible Box"（弹性盒子）。它的核心思想是：容器（flex container）定义主轴方向，子元素（flex items）自动填充可用空间，或者在空间不足时自动收缩。

这就解决了一个经典问题：如何让三个按钮等宽分布？在浮动布局时代，你需要给每个按钮设 width: 33.333%。但如果增加第四个按钮，就要改成 25%。用 Flexbox，只需 flex: 1;，每个子元素自动均分空间——加多少个都能自适应。

1.4.2 我们的主页用了哪些 Flexbox 属性？

.home-wrap {
  display: flex;
  flex-direction: column;      /* 主轴方向：从上到下 */
  align-items: center;         /* 交叉轴对齐：居中 */
  justify-content: center;     /* 主轴对齐：居中 */
  min-height: 100vh;           /* 最小高度 = 视口高度 */
}

.home-entries {
  display: flex;
  gap: 20px;                   /* 子元素间距 */
  justify-content: center;     /* 水平居中 */
  flex-wrap: wrap;             /* 空间不够时换行 */
}

详解：

  • display: flex：开启 Flexbox 布局。子元素自动变成 flex items。
  • flex-direction: column：主轴方向为垂直（从上到下）。默认是 row（水平从左到右）。
  • justify-content: center：主轴上的对齐方式。column 模式下主轴是垂直方向，所以这是"垂直居中"。
  • align-items: center：交叉轴上的对齐方式。column 模式下交叉轴是水平方向，所以这是"水平居中"。
  • min-height: 100vh：最小高度为视口高度的 100%。vh 是 viewport height 的缩写，1vh = 视口高度的 1%。这样即使内容很少，页面也会占满整个屏幕。
  • gap: 20px：子元素之间的间距。这个属性是 Flexbox 和 Grid 共用的，比老式的 margin 方案简洁多了。
  • flex-wrap: wrap：空间不够时换行。没有它，三个卡片会在窄屏幕上被挤变形。

🤔 为什么 justify-content 和 align-items 容易搞混？

这是很多初学者的痛点。记忆口诀：

justify 管主轴，align 管交叉轴。

当 flex-direction 是 row（默认值）时：主轴 = 水平，交叉轴 = 垂直。所以 justify-content 控制水平对齐，align-items 控制垂直对齐。

当 flex-direction 是 column 时：主轴 = 垂直，交叉轴 = 水平。所以 justify-content 控制垂直对齐，align-items 控制水平对齐。

这个命名来自早期排版术语。在活字印刷时代，"justify" 指的是调整字间距让文字两端对齐。在 CSS 中，它被扩展为"沿主轴分布"。

如果你觉得太难记，可以用一个更直观的方法：把 justify-content 想成"justify = 排布"，align-items 想成"align = 对齐"。主轴上怎么排布，交叉轴上怎么对齐。

1.5 颜色与视觉设计——为什么选择"暗金"配色？

1.5.1 配色方案分析

我们的主页使用了以下配色：

用途色值RGB感觉
背景#0c0c0e接近纯黑，略带蓝深邃、沉稳、高级感
主标题#c9a96e暗金色典雅、品质、文化感
正文#f0ece6暖白柔和、不刺眼
次要文字#a09a90灰棕低调、层次

📖 故事时间：暗金配色的文化渊源

"暗金"配色不是随意选的。黑色 + 金色的组合在人类审美中有着悠久的历史：

  • 古埃及：法老的棺椁用黑曜石和黄金装饰，象征永恒与神圣
  • 中世纪欧洲：宗教画用金箔贴背景，黑色天鹅绒边框，营造庄严感
  • 现代奢侈品：GUCCI、Dolce & Gabbana 的经典配色就是黑金
  • Apple 的暗色模式：2019 年 iOS 13 推出 Dark Mode，黑色背景 + 金色/橙色强调色成为高端 UI 的趋势

为什么选择暗色背景而不是白色？

第一，暗色模式更护眼。白色屏幕的亮度（尤其是蓝光）会刺激视网膜，长时间阅读容易疲劳。暗色背景减少了光线进入眼睛的量。

第二，暗色模式更省电。OLED 屏幕（iPhone 12 及之后的机型、大部分安卓旗舰）在显示黑色时，像素是完全关闭的，几乎不耗电。

第三，暗色背景让内容更突出。白色背景上，所有元素都在"发光"；暗色背景上，只有你想突出的元素在发光，其他元素自然隐入背景。这种"光与暗的对比"天然具有戏剧感和高级感。

至于为什么选择金色而不是红色、蓝色？金色在人类文化中象征着"珍贵""成就""品质"。给网站用金色，潜意识里传达了"这个网站是有价值的"的信息。红色太激烈，蓝色太冰冷，绿色太活泼——对于"记录生活、沉淀思考"的网站来说，金色刚刚好。

1.5.2 颜色的技术实现

CSS 中颜色可以用多种方式表示：

格式示例适用场景
十六进制#0c0c0e最常用，简洁
RGBrgb(12, 12, 14)需要计算透明度时
RGBArgba(201, 169, 110, 0.5)半透明颜色
HSLhsl(38, 43%, 61%)需要调整亮度/饱和度时
颜色名black, gold, red学习时快速测试

我们主要用十六进制，因为它最简洁。#c9a96e 展开就是 rgb(201, 169, 110)。

rgba(255, 255, 255, 0.02) 是什么意思？

RGBA 的第四个参数是 alpha（透明度），范围 0（完全透明）到 1（完全不透明）。0.02 表示几乎透明，只保留一点点白色调。这用来做卡片背景——比纯黑背景稍微亮一点点，但不明显，营造出微妙的层次感。

🤔 为什么卡片背景用 rgba 而不是 solid color？

如果用纯灰色（比如 #1a1a1a）做卡片背景，它就是一个固定的颜色。但如果以后你想换主背景色，卡片背景也要跟着改。

用 rgba(255,255,255,0.02) 的好处是：它是"在当前背景上加一点点白色"。无论背景色是什么（黑色、深蓝、深红），卡片都会比背景稍微亮一点点。这种"相对亮度"的设计比"绝对颜色"更灵活。

这也是为什么很多现代 UI 框架（比如 Tailwind CSS）推荐用 opacity 控制层次，而不是用固定的灰色。opacity 是"叠加在背景上的效果"，而固定灰色是"独立于背景的颜色"。

1.6 过渡动画——让页面"活"起来

1.6.1 transition 属性详解

.entry-card {
  transition: all 0.3s ease;
}

这行代码的意思是：当 .entry-card 的任何 CSS 属性发生变化时，用 0.3 秒的时间平滑过渡，使用 ease 时间函数。

展开写法：

.entry-card {
  transition-property: all;        /* 哪些属性要过渡 */
  transition-duration: 0.3s;       /* 过渡时间 */
  transition-timing-function: ease; /* 时间曲线 */
  transition-delay: 0s;            /* 延迟多久开始 */
}

transition-property：指定哪些属性变化时有动画。可以是 all（所有属性），也可以是具体的属性名列表（transition: transform 0.3s, border-color 0.5s）。

transition-duration：动画持续时间。太短（0.1s）用户可能注意不到，太长（1s）会让人觉得卡顿。0.3s 是 UI 动画的"甜点值"——足够让用户感知到变化，又不会拖慢操作。

transition-timing-function：速度曲线。

值贝塞尔曲线效果
linear匀速直线从头到尾速度一样，机械感
ease先快后慢默认，最自然
ease-in慢-快加速，像物体下落
ease-out快-慢减速，像物体滑入
ease-in-out慢-快-慢对称，像钟摆

📖 故事时间：贝塞尔曲线与迪士尼动画

ease 背后的数学是贝塞尔曲线（Bézier Curve）。这是一种由法国工程师 Pierre Bézier 在 1960 年代为雷诺汽车公司发明的曲线，用来设计汽车外形。

贝塞尔曲线的特点是：通过几个"控制点"就能定义一条平滑的曲线。在 CSS 中，cubic-bezier(0.25, 0.1, 0.25, 1.0) 这四个数字就是两个控制点的坐标。

迪士尼动画师们在 1930 年代总结出的"动画十二原则"中，有一条叫"缓入缓出"（Slow In and Slow Out）：真实的运动不会是匀速的，加速和减速让动画更有质感。CSS 的 ease 函数本质上就是这条原则的数学实现。

这就是为什么 ease 看起来比 linear 更舒服——linear 太机械了，像机器人；ease 有加速和减速，像自然界的运动。

1.6.2 我们的 hover 效果

.entry-card:hover {
  transform: scale(1.08);
  border-color: rgba(201, 169, 110, 0.4);
  box-shadow: 0 0 30px rgba(201, 169, 110, 0.1);
}

逐行详解：

transform: scale(1.08)：缩放变换。scale(1.08) 表示放大到原来的 108%。为什么是 1.08 而不是 1.1？

UI 设计中的缩放有一个"刚刚好"的区间：1.05（5%）太不明显，用户可能注意不到；1.15（15%）太夸张，会破坏布局。1.08（8%）是一个经过大量 A/B 测试验证的"甜点值"——足够引起注意，又不会过于突兀。

border-color: rgba(201, 169, 110, 0.4)：边框颜色变成金色，但透明度只有 40%。为什么不直接用 #c9a96e（不透明金色）？

因为不透明的金色边框会很刺眼，像一个发光的框。半透明的效果更柔和，暗示"被激活了"但不抢夺注意力。这是"减法设计"的理念——不要给用户太多刺激。

box-shadow: 0 0 30px rgba(201, 169, 110, 0.1)：外发光效果。

box-shadow 的语法是 x偏移 y偏移 模糊半径 颜色。这里 x 和 y 都是 0，表示光源在正上方，向四周均匀扩散。30px 是模糊半径——越大越柔和。颜色是 10% 透明度的金色。

为什么不加 spread（扩展半径，第四个数字）？因为加了 spread 会让阴影边缘变硬，失去"光晕"的柔和感。我们的目标是营造"泛金光"的氛围，而不是"有一个金色的框"。

🤔 为什么:hover 要同时改变三个属性？

如果只改变一个属性（比如只放大），效果会很单调。同时改变三个属性（大小、边框、阴影）会形成"层次丰富的反馈"：

  • 放大 → 告诉用户"这个元素被激活了"
  • 金色边框 → 告诉用户"这是当前焦点"
  • 光晕 → 告诉用户"这个元素很重要"

这三个信号叠加在一起，给用户一个完整的"这个卡片可以点击"的认知。这是视觉设计中的"冗余编码"原则——用多种方式传达同一个信息，确保用户不会错过。

但也要避免过度设计。如果再加一个旋转、一个颜色变化、一个文字变大，就会信息过载，让用户感到混乱。三个属性是"刚好足够"的阈值。

1.7 响应式设计——为什么你的网站要在手机上也能看？

1.7.1 为什么必须做响应式？

2016年是一个转折点：全球移动互联网流量首次超过了桌面互联网流量。今天，超过 60% 的网页浏览发生在手机上。

如果你的网站在手机上显示错乱（文字太小、需要横向滚动、按钮点不到），你会失去大多数访客。

📖 故事时间：响应式设计的诞生

2010年，设计师 Ethan Marcotte 在 A List Apart 杂志上发表了一篇文章《Responsive Web Design》（响应式网页设计）。他提出了三个核心技术：

  1. 流式布局（Fluid Grids）：用百分比而不是固定像素定义宽度
  2. 弹性图片（Flexible Images）：图片最大宽度不超过容器宽度
  3. 媒体查询（Media Queries）：根据屏幕宽度应用不同的 CSS

这篇文章彻底改变了网页设计。在此之前，常见的做法是"桌面版 + 移动版"两个网站（比如 m.taobao.com）。维护两个网站成本极高，而且内容同步困难。响应式设计让"一个网站适配所有设备"成为可能。

Ethan 后来把响应式设计的理念写成了一本书《Responsive Web Design》，成为了这个领域的圣经。今天，响应式设计不再是"可选项"，而是"必须项"。Google 的搜索排名算法会惩罚没有移动适配的网站。

1.7.2 我们的响应式策略

我们的主页没有做复杂的媒体查询，但有几个天然的响应式特性：

  • flex-wrap: wrap：三个卡片在窄屏幕上自动换行，不会挤在一起
  • min-height: 100vh：内容始终占满屏幕高度，不会留空白
  • padding: 40px 20px：左右 20px 的 padding 确保文字不会贴边

这种"移动优先"的设计思路是：先确保在小屏幕上能看，然后再考虑大屏幕上的优化。这比"先设计桌面版再砍功能到手机版"更合理。

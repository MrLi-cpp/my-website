第五章：用户系统——注册、登录、安全

本章目标：实现用户注册和登录功能。密码要加密存储，登录后服务器要"记住"用户身份。这是网站从"匿名"走向"身份化"的关键一步。

学完后你能做什么：可以注册账号、登录、退出。登录后才能发表朋友圈，而且能看到"当前登录用户"。你会理解密码加密、Session 会话、HTTP Cookie 这些核心安全概念。

本章难点：理解"哈希"和"加密"的区别；理解 Session 和 Cookie 如何配合工作；理解 bcrypt 的"慢"为什么是安全特性。

5.1 为什么密码不能明文存储？

第四章的朋友圈允许任何人发表，没有身份验证。这在真实网站里是不可接受的——你需要知道每条动态是谁发的，而且不能让陌生人冒充你。

用户系统的核心是"身份认证"：证明"你是你"。最常用的方式是"用户名 + 密码"。

5.1.1 身份认证的演进

计算机系统的身份认证经历了几个阶段：

第一阶段（1960s）：无认证。 早期的多用户系统（如 CTSS）没有密码概念，任何人都能登录任何账号。这很快导致了混乱——学生删除了教授的文件，员工篡改了工资记录。

第二阶段（1970s）：密码文件。 UNIX 系统引入了 /etc/passwd 文件，存储用户名和密码。但密码是明文存储的，任何能读取这个文件的人都能看到所有密码。1974 年，Robert Morris 发明了密码哈希（crypt），把密码变成不可逆的字符串。这个算法成为了 UNIX 密码存储的标准。

第三阶段（1990s+）：加盐哈希。 人们发现单纯的哈希也有问题——可以用"彩虹表"（预计算的常见密码哈希值）快速破解。于是"盐值"（salt）被引入：在密码前加一个随机字符串再哈希，让彩虹表失效。

第四阶段（2010s+）：自适应哈希。 计算机速度越来越快，即使加盐的哈希也能被暴力破解。bcrypt、scrypt、Argon2 等"自适应"算法出现了——它们可以调节计算强度，让哈希速度永远慢于硬件提升速度。

📖 故事时间：明文密码的血泪史

2011 年，中国程序员社区 CSDN 发生了大规模用户数据泄露——600 万用户的用户名和密码被公开。最令人震惊的是：这些密码都是明文存储的，没有任何加密。

这意味着攻击者直接拿到了用户的原始密码。更可怕的是，很多人在多个网站用同一个密码，所以 CSDN 的泄露导致这些用户的 Gmail、QQ、网银账号也面临风险。

CSDN 事件后，"密码必须加密存储"成为了行业铁律。今天，任何存储明文密码的网站都会被视为严重安全事故。2016 年，LinkedIn 泄露了 1.17 亿条密码哈希（但不是明文），但由于用的是 SHA-1（已被破解），90% 的密码在几天内就被还原了。

另一个著名的案例是 2013 年的 Adobe 泄露——1.5 亿用户数据被盗。Adobe 用的是弱加密（ECB 模式的 3DES），而且密钥管理不善。这个事件让 Adobe 损失了数千万美元的诉讼赔偿，CEO 引咎辞职。

密码安全的三条铁律：1）永远不明文存储；2）使用行业标准的哈希算法；3）加盐防止彩虹表攻击。

5.2 bcrypt——密码哈希的行业标准

5.2.1 哈希 vs 加密

很多人把"哈希"和"加密"混为一谈，但它们是完全不同的概念：

特性哈希（Hash）加密（Encryption）
是否可逆否（单向）是（可解密）
用途验证数据完整性保护数据机密性
密码存储正确方式错误方式
常见算法bcrypt, Argon2, SHA-256AES, RSA, ChaCha20
相同输入输出总是相同（确定性）取决于密钥

密码存储应该用哈希，因为：服务器不需要知道用户的原始密码是什么，只需要验证"用户输入的密码，哈希后是否和数据库里存的哈希值一致"。

如果用加密，服务器需要保存解密密钥。一旦密钥泄露，所有密码都能被解密。而且加密是可逆的——这意味着理论上可以还原出原始密码。哈希是不可逆的，即使数据库泄露，攻击者也无法还原密码（只能暴力尝试）。

5.2.2 为什么选择 bcrypt？

常见的密码哈希算法有：

算法特点是否推荐
MD5速度快，128 位输出，已被破解否（过时）
SHA-1速度快，160 位输出，已被破解否（过时）
SHA-256速度快，256 位输出，需要加盐+多次迭代才能用于密码可用但需额外配置
bcrypt自带盐值，可调节计算强度，192 位输出推荐
scrypt内存困难型，抗 GPU/ASIC 破解推荐
Argon22015 年密码哈希竞赛冠军，可调内存和时间参数推荐（但 bcrypt 更普及）

bcrypt 的设计者是 Niels Provos 和 David Mazières，基于 Bruce Schneier 的 Blowfish 密码算法改编。bcrypt 的核心特点是"慢"——它故意让哈希计算耗时几百毫秒，这样即使攻击者拿到了哈希值，暴力破解也需要极长时间。

🤔 为什么密码哈希要"慢"？

这和普通哈希的需求正好相反。MD5/SHA 设计目标是"快"——计算文件哈希时需要快速处理大量数据。SHA-256 的计算速度是每秒数百万次。

但密码哈希的需求是"抗暴力破解"。如果哈希计算只需要 1 微秒，攻击者一秒钟就能试 100 万个密码。如果哈希计算需要 200 毫秒，攻击者一秒钟只能试 5 个密码。

假设一个网站有 1000 万用户，攻击者拿到数据库后：

  • 用 MD5（1 微秒/次）：试完所有用户的 Top 100 常用密码只需要 1000 秒（约 17 分钟）
  • 用 bcrypt cost=10（200 毫秒/次）：试完同样的密码需要 2.3 年

bcrypt 的"慢"是通过"成本因子"（cost factor）控制的。默认值 10 表示计算要执行 2^10 = 1024 轮。你可以提高到 12 或 14，让计算更慢、更安全（但也更消耗服务器资源）。

bcrypt 的另一个特点是"自带盐值"（salt）。每次哈希时，bcrypt 会生成一个 16 字节的随机盐值，拼接到密码前面再哈希。这意味着：即使两个用户的密码相同，存储的哈希值也不同。这防止了"彩虹表攻击"（预计算常见密码的哈希值）。

bcrypt 的输出格式是：$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy。其中 $2a$ 是版本标识，10 是成本因子，后面 22 个字符是盐值，最后 31 个字符是哈希值。

📖 故事时间：密码哈希竞赛

2012 年，密码学研究社区发起了一场公开竞赛——"Password Hashing Competition"，目标是选出下一代密码哈希标准。参赛算法包括 Argon2、Catena、Lyra2、Makwa、yescrypt。

2015 年，Argon2 获胜。它的设计者 Alex Biryukov 和 Daniel Dinu 来自卢森堡大学。Argon2 有两种模式：Argon2d（抵抗 GPU 破解）和 Argon2i（抵抗侧信道攻击）。

但 bcrypt 并没有被淘汰——它已经运行了 20 多年，被广泛验证为安全。很多安全专家建议：如果你的系统已经在用 bcrypt，没有必要为了"最新"而迁移到 Argon2。bcrypt 仍然足够安全，而且生态系统更成熟。

Wikipedia 就是一个例子：它用 PBKDF2（另一种慢哈希）存储用户密码，而不是 bcrypt 或 Argon2。选择哪种算法取决于你的威胁模型——对于个人网站，bcrypt 完全够用。

5.2.3 安装 bcryptjs

步骤
cd ~/Desktop/my-website
npm install bcryptjs

我们用 bcryptjs 而不是 bcrypt，因为 bcryptjs 是纯 JavaScript 实现，不需要编译原生 C++ 扩展。bcrypt 在某些系统上安装时会因为缺少编译工具（Python、node-gyp、C++ 编译器）而失败，bcryptjs 避免了这个问题。

bcryptjs 的 API 和 bcrypt 完全一致，可以无缝替换。唯一的缺点是速度比原生 bcrypt 慢约 30%——但对于个人网站的注册/登录频率，这完全不是问题。

5.3 Session——服务器如何"记住"登录状态

5.3.1 HTTP 是无状态的

HTTP 协议的一个核心特性是无状态（Stateless）——每个请求都是独立的，服务器不会记住上一个请求是谁发的。

这意味着：你登录时发送了用户名和密码，服务器验证通过后说"好的你是霁光"。下一秒你请求"发表动态"，服务器又忘了你是谁，需要你重新证明身份。

显然这不可接受。解决方案是Session（会话）：服务器生成一个唯一的"会话 ID"，发给浏览器。浏览器每次请求都带上这个 ID，服务器根据 ID 查出"这个会话属于哪个用户"。

📖 故事时间：Session 和 Cookie 的发明

1994 年，Netscape 的程序员 Lou Montulli 发明了 Cookie。他的初衷是解决一个购物网站的问题：用户把商品加入购物车后，刷新页面购物车就清空了——因为服务器记不住。

Cookie 的设计很简单：服务器在响应头里发送 Set-Cookie，浏览器保存后，后续请求的 Cookie 头里自动带上这个值。

Session 是在 Cookie 基础上的抽象：服务器生成一个随机字符串（Session ID），通过 Cookie 发给浏览器。服务器端用一个"Session 存储"（内存、文件、Redis、数据库）来保存 Session ID 和用户数据的映射。

Lou Montulli 后来回忆说，他没想到 Cookie 会被用于追踪用户、投放广告。"如果当时知道，我可能会设计得更谨慎。"但 Cookie 已经成为 Web 的基石，没有它，登录、购物车、个性化推荐都无法实现。

2011 年，欧盟通过了一项法规（Cookie Directive），要求网站在使用 Cookie 前必须获得用户同意。这就是为什么你今天访问很多网站时会看到一个"我们使用 Cookie"的弹窗。

5.3.2 Session 的工作流程

整个 Session 流程涉及三方：浏览器、服务器、Session 存储。

┌─────────┐                    ┌─────────┐              ┌─────────────┐
│ 浏览器   │◄───── ① 首次请求 ───►│ 服务器  │             │ Session 存储 │
│         │                    │         │             │（内存/文件）  │
│         │◄──── ② 分配 Session ID ─│         │             │             │
│ 保存    │     Set-Cookie: sid=xxx  │ 创建记录  │─────►│ sid=xxx     │
│ Cookie  │                    │ userId=1 │             │ userId=1    │
│         │───── ③ 后续请求 ─────►│         │             │             │
│ Cookie: │    Cookie: sid=xxx    │ 查 sid   │─────►│ 返回 userId │
│ sid=xxx │◄──── ④ 返回数据 ─────│ 知道用户  │◄─────│             │
└─────────┘                    └─────────┘              └─────────────┘

步骤解析：

  1. 用户第一次访问网站，服务器发现请求里没有 Session ID
  2. 服务器创建一个新的 Session 记录（包含 userId 等信息），生成一个唯一的 Session ID
  3. 服务器通过 HTTP 响应头的 Set-Cookie 字段，把 Session ID 发给浏览器
  4. 浏览器保存 Cookie，后续每次请求都自动带上
  5. 服务器收到请求后，从 Cookie 中取出 Session ID，去 Session 存储里查对应的用户数据

5.3.3 Cookie 的属性

Session ID 通过 Cookie 传递。Cookie 有很多安全属性：

属性作用建议值
HttpOnly禁止 JavaScript 读取 Cookietrue（防止 XSS 窃取）
Secure只在 HTTPS 连接中发送 Cookietrue（生产环境）
SameSite控制跨站请求时是否发送 Cookielax 或 strict
MaxAgeCookie 有效期（毫秒）7 * 24 * 60 * 60 * 1000（7天）
PathCookie 生效的路径/（全站）

HttpOnly 是最关键的安全属性。如果没有它，恶意脚本可以通过 document.cookie 读取 Session ID，然后冒充用户身份。

🤔 为什么需要 SameSite？

SameSite 是为了防止 CSRF（跨站请求伪造）攻击。攻击者可以在第三方网站上放一个图片或表单，指向你的网站的 API：

<!-- 攻击者的网站 -->
<img src="https://你的网站.com/api/posts?content=我是垃圾广告">

如果用户已经登录了你的网站，浏览器会自动带上 Cookie 发送这个请求，导致在用户不知情的情况下发表了动态。

SameSite=lax 表示：只有在用户从外部网站点击链接（GET 请求）时才发送 Cookie，而 img、iframe、POST 表单不会发送。这阻止了大部分 CSRF 攻击。

SameSite=strict 更安全——即使是点击链接也不发送 Cookie，只有当用户在地址栏直接输入 URL 时才发送。但这会影响用户体验（从外部链接进入网站时需要重新登录）。

5.4 安装 express-session

步骤
npm install express-session session-file-store

express-session 是 Express 官方的 Session 中间件。session-file-store 是一个"Session 存储驱动"——它把 Session 数据保存在服务器的文件系统里（而不是内存里，这样服务器重启后 Session 不会丢失）。

Session 存储有多种选择：

存储方式优点缺点
内存（默认）最快服务器重启丢失；多进程不共享
文件持久化；零配置慢；文件系统限制
Redis快；分布式共享需要安装 Redis
数据库持久化；结构化查询慢；增加数据库负担
MongoDBNoSQL；灵活需要 MongoDB

对于个人网站，文件存储完全够用。当用户量增大时，可以迁移到 Redis。

5.5 用户系统代码实现

5.5.1 创建 users 表

在数据库初始化时添加 users 表：

db.run(`
  CREATE TABLE IF NOT EXISTS users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    is_admin INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
  )
`);

UNIQUE 表示用户名不能重复。如果尝试注册一个已存在的用户名，SQLite 会报错。

is_admin INTEGER DEFAULT 0：0 表示普通用户，1 表示管理员。默认都是普通用户。

5.5.2 注册接口

const bcrypt = require('bcryptjs');

app.post('/api/register', async (req, res) => {
  const { username, password } = req.body;
  
  // 验证输入
  if (!username || !password) {
    return res.status(400).json({ error: '用户名和密码不能为空' });
  }
  
  if (username.length < 3 || username.length > 20) {
    return res.status(400).json({ error: '用户名长度 3-20 个字符' });
  }
  
  if (password.length < 6) {
    return res.status(400).json({ error: '密码至少 6 位' });
  }
  
  try {
    // 生成密码哈希（成本因子 10）
    const password_hash = await bcrypt.hash(password, 10);
    
    // 插入数据库
    db.run(
      'INSERT INTO users (username, password_hash) VALUES (?, ?)',
      [username, password_hash],
      function(err) {
        if (err) {
          // 用户名已存在（UNIQUE 约束冲突）
          if (err.message.includes('UNIQUE constraint failed')) {
            return res.status(409).json({ error: '用户名已被注册' });
          }
          return res.status(500).json({ error: err.message });
        }
        
        res.json({ message: '注册成功', username });
      }
    );
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

逐行解析：

const password_hash = await bcrypt.hash(password, 10);

  • bcrypt.hash 是异步函数（需要 await），因为哈希计算需要时间
  • 第一个参数是原始密码（明文）
  • 第二个参数是成本因子（cost factor），10 表示 2^10 = 1024 轮迭代
  • 返回的 password_hash 是一个字符串，可以直接存入数据库

if (err.message.includes('UNIQUE constraint failed'))

  • SQLite 的 UNIQUE 约束冲突时，错误消息包含"UNIQUE constraint failed"
  • 我们根据这个来判断是否是用户名重复
  • 返回 409 Conflict 状态码，这是 HTTP 中表示"资源冲突"的标准码

5.5.3 登录接口

app.post('/api/login', (req, res) => {
  const { username, password } = req.body;
  
  if (!username || !password) {
    return res.status(400).json({ error: '用户名和密码不能为空' });
  }
  
  db.get(
    'SELECT * FROM users WHERE username = ?',
    [username],
    async (err, user) => {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      
      // 用户不存在
      if (!user) {
        return res.status(401).json({ error: '用户名或密码错误' });
      }
      
      // 验证密码
      const isValid = await bcrypt.compare(password, user.password_hash);
      
      if (!isValid) {
        return res.status(401).json({ error: '用户名或密码错误' });
      }
      
      // 登录成功，写入 Session
      req.session.userId = user.id;
      req.session.username = user.username;
      
      res.json({ message: '登录成功', username: user.username });
    }
  );
});

逐行解析：

db.get('SELECT * FROM users WHERE username = ?', [username], ...)

  • db.get 只返回一行结果（适合查询单条记录）
  • 用 ? 占位符防止 SQL 注入
  • callback 是 async 函数，因为里面要用 await bcrypt.compare

await bcrypt.compare(password, user.password_hash)

  • bcrypt.compare 比较明文密码和哈希值
  • 注意参数顺序：第一个是明文，第二个是哈希。如果反过来，永远返回 false
  • bcrypt 会自动提取哈希中的盐值和成本因子，用同样的方式计算后再比较

req.session.userId = user.id;

  • express-session 在 req 对象上添加了 session 属性
  • 把用户 ID 和用户名存入 Session，后续请求都能读取
  • 这些数据被保存在 Session 存储中（我们的配置是文件存储）

🤔 为什么登录失败时不告诉用户是"用户名不存在"还是"密码错误"？

你可能会想：如果用户不存在，返回"用户名不存在"；如果密码错了，返回"密码错误"——这样用户就知道问题出在哪了。

但安全专家不同意这种做法。原因：攻击者可以利用这个信息来"枚举"用户名。他们批量尝试常见用户名，如果系统返回"用户名不存在"，就知道这个用户名没被注册；如果返回"密码错误"，就知道用户名存在，只需要猜密码。

通过统一返回"用户名或密码错误"，攻击者无法区分是用户名不存在还是密码错了，大大增加了枚举成本。这是安全设计中的"信息隐藏"原则——不要给攻击者提供额外的信息。

类似的原则还有：注册时不应该告诉用户"这个用户名已被注册"（虽然很多网站这么做），因为这也会泄露用户名是否存在。理想的做法是：让用户选择"登录或注册"，或者注册成功后直接发一封验证邮件。

5.5.4 Session 中间件配置

const session = require('express-session');
const FileStore = require('session-file-store')(session);

app.use(session({
  store: new FileStore({ 
    path: './sessions',     // Session 文件保存路径
    retries: 0            // 写入失败不重试
  }),  
  secret: 'your-secret-key-here',  // 签名密钥
  resave: false,                    // 不强制保存未修改的 Session
  saveUninitialized: false,          // 不保存空 Session
  name: 'sessionId',                // Cookie 名称（默认 connect.sid）
  cookie: {
    maxAge: 7 * 24 * 60 * 60 * 1000,  // 7 天有效期
    httpOnly: true,                    // 禁止 JavaScript 读取
    secure: false,                     // 开发环境用 http
    sameSite: 'lax'                    // 防止 CSRF
  }
}));

配置解析：

配置项说明
storeSession 存储方式。FileStore 存文件，RedisStore 存 Redis
secret用于签名 Session ID 的密钥。生产环境应该是随机长字符串
resavefalse 表示 Session 没修改时不保存（减少 I/O）
saveUninitializedfalse 表示未登录用户不创建 Session（减少垃圾 Session）
nameCookie 名称。改成 sessionId 可以隐藏使用的框架
cookie.maxAgeSession 有效期。过期后浏览器删除 Cookie
cookie.httpOnlytrue 阻止 XSS 脚本读取 Cookie
cookie.securetrue 只在 HTTPS 发送（开发环境设 false）
cookie.sameSitelax 防止 CSRF，同时保留正常跳转体验
⚠️ 重要提示
生产环境中，secret 绝不能写在代码里！应该放在环境变量中： 
secret: process.env.SESSION_SECRET || 'fallback-secret'

5.5.5 获取当前用户接口

app.get('/api/me', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '未登录' });
  }
  
  res.json({
    id: req.session.userId,
    username: req.session.username
  });
});

前端可以用这个接口判断用户是否登录，并显示用户名。

5.5.6 退出登录

app.post('/api/logout', (req, res) => {
  req.session.destroy();
  res.json({ message: '已退出登录' });
});

req.session.destroy() 删除服务器端的 Session 记录，并通知浏览器删除 Cookie。下次请求时，浏览器不再发送 Session ID。

5.7 密码安全的完整流程

让我们回顾整个密码安全流程，确保每个环节都正确：

注册流程：
1. 用户输入密码 → 前端验证长度（≥6位）
2. 发送到服务器 → 服务器再次验证
3. bcrypt.hash(密码, 10) → 生成 $2a$10$... 格式的哈希
4. 存入数据库 → 只存哈希，不存明文

登录流程：
1. 用户输入密码 → 发送到服务器
2. 查数据库获取 password_hash
3. bcrypt.compare(输入的密码, password_hash)
4. 比较结果：true → 创建 Session；false → 返回错误

验证流程：
1. 浏览器发送请求 → 自动带上 Cookie: sessionId=xxx
2. 服务器查 Session 存储 → 获取 userId
3. 根据 userId 判断用户身份和权限

📝 练习时间

  1. 尝试用 bcrypt 哈希同一个密码两次，观察两次输出是否相同（应该不同，因为盐值随机）
  2. 把成本因子从 10 改成 4，感受哈希速度的变化
  3. 在浏览器 DevTools 的 Application → Cookies 面板中查看 sessionId Cookie
  4. 尝试用 JavaScript 读取 document.cookie，验证 httpOnly 生效
  5. 把 resave 改成 true，观察 sessions 目录下文件数量变化

5.8 常见问题急救站

问题现象原因解决方案
Session 不持久 刷新页面后登录状态丢失 Session 存在内存里，服务器重启丢失 使用 session-file-store 或 Redis
密码验证失败 确认密码正确但登录失败 bcrypt.compare 的参数顺序错了 必须是 bcrypt.compare(明文, 哈希)，不能反过来
Cookie 不发送 登录后 /api/me 返回未登录 跨域或 secure/cookie 配置错误 开发环境 secure: false，sameSite: 'lax'
用户名重复报错 注册时报 SQLITE_CONSTRAINT_UNIQUE UNIQUE 约束冲突 在注册时先查重，或捕获错误返回友好提示
bcrypt 太慢 注册/登录响应超过 1 秒 bcrypt 哈希计算耗时 成本因子从 10 降到 8（开发时），生产保持 10+
session 文件堆积 sessions 目录下文件越来越多 空 Session 被创建后未清理 saveUninitialized: false；定期清理过期文件
XSS 窃取 Cookie 虽然设置了 httpOnly 但担心不够 httpOnly 是最强防御，但还有其他层面 同时开启 secure、sameSite；做 CSP 策略
生产环境密钥泄露 GitHub 上代码暴露了 secret 敏感信息写死在代码里 用 .env 文件 + dotenv；把 .env 加入 .gitignore

5.9 知识点回顾

📚 本章核心概念回顾

  • 哈希（Hash）：单向函数，不可逆。用于密码存储验证。
  • 加密（Encryption）：双向函数，可解密。用于数据传输保护。
  • bcrypt：自适应密码哈希算法，自带盐值，可调节计算强度。
  • 盐值（Salt）：随机字符串，拼接到密码前再哈希。防止彩虹表攻击。
  • Session：服务器端存储的用户会话数据，用 Session ID 标识。
  • Cookie：浏览器存储的小型文本，自动在请求中发送给服务器。
  • httpOnly：Cookie 属性，禁止 JavaScript 读取，防止 XSS 窃取。
  • sameSite：Cookie 属性，控制跨站请求时是否发送，防止 CSRF。
  • SQL 注入：通过恶意输入篡改 SQL 语句。用 ? 占位符防御。

第七章：博客系统——Markdown 渲染与文章管理

本章目标：实现博客功能。用户可以发布文章，访客可以浏览文章列表和详情。文章支持 Markdown 格式渲染。

学完后你能做什么：有一个完整的博客系统——列表页展示所有文章，详情页渲染 Markdown 格式的文章内容，支持浏览计数和标签筛选。

7.1 为什么博客需要 Markdown？

HTML 是网页的标记语言，但直接写 HTML 很繁琐：

<h1>标题</h1>
<p>这是<strong>加粗</strong>的文字</p>
<ul>
  <li>列表项1</li>
  <li>列表项2</li>
</ul>

Markdown 是一种轻量级标记语言，用更简洁的语法达到同样的效果：

# 标题

这是**加粗**的文字

- 列表项1
- 列表项2

Markdown 让你专注于内容写作，而不是 HTML 标签。

📖 故事时间：Markdown 的诞生

2004 年，John Gruber 和 Aaron Swartz 合作创造了 Markdown。Gruber 是一位科技博主，他对当时流行的"用 HTML 写博客"很不满意——写作时要在内容和标签之间来回切换，思路经常被打断。

Markdown 的设计目标是："用一种易读易写的纯文本格式，然后转换成有效的 HTML。"它的语法参考了电子邮件中常用的纯文本标记习惯：

  • *星号包围* = 斜体
  • **双星号** = 加粗
  • > 大于号 = 引用
  • `反引号` = 代码

2012 年，GitHub 推出了 GitHub Flavored Markdown（GFM），增加了表格、任务列表、代码高亮等扩展。今天，Markdown 是程序员写文档、博客、README 的标准格式。GitHub、Notion、Obsidian、Typora 都支持 Markdown。

有趣的是，Markdown 至今没有官方标准。John Gruber 只定义了最初的语法规范，但没有标准化组织维护它。这导致不同平台对 Markdown 的解析略有差异。CommonMark（2014 年）试图解决这个问题，但 GFM 等扩展仍然是各平台自己的实现。

Aaron Swartz 是一位天才程序员和互联网活动家。他参与了 RSS 1.0 规范、Creative Commons 技术框架、 Reddit 的早期开发。但他最著名的事迹是 2011 年通过 MIT 网络下载了 JSTOR 学术数据库的数百万篇论文，被美国政府以"计算机欺诈"起诉。面对 35 年监禁和 100 万美元罚款的威胁，Aaron 在 2013 年 1 月自杀，年仅 26 岁。他的死引发了关于学术开放获取和计算机犯罪法律边界的广泛讨论。

7.3 数据库设计——blogs 表

7.3.1 表结构

CREATE TABLE IF NOT EXISTS blogs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  summary TEXT,
  cover_image TEXT,
  tags TEXT,
  user_id INTEGER NOT NULL,
  view_count INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

字段说明：

字段说明用途
title文章标题列表页和详情页显示
content文章内容Markdown 格式的正文
summary摘要列表页显示，不显示全文
cover_image封面图 URL列表页的缩略图
tags标签逗号分隔，如 "技术,前端,教程"
user_id作者 ID外键，关联 users 表
view_count浏览次数统计文章阅读量
updated_at最后更新时间显示文章是否近期更新

summary 字段的设计思路：列表页不需要显示全文，只需要一段摘要。如果没有 summary，可以用 content 的前 100 字自动截取。

7.4 后端 API

7.4.1 发布文章

app.post('/api/blogs', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  const { title, content, summary, tags } = req.body;
  
  if (!title || !content) {
    return res.status(400).json({ error: '标题和内容不能为空' });
  }
  
  db.run(
    `INSERT INTO blogs (title, content, summary, tags, user_id)
     VALUES (?, ?, ?, ?, ?)`,
    [title, content, summary || '', tags || '', req.session.userId],
    function(err) {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      res.json({ id: this.lastID, message: '发布成功' });
    }
  );
});

summary || '' 表示：如果 summary 没传，用空字符串。这是 JavaScript 的"逻辑或"运算符——当 summary 为 undefined、null 或空字符串时，返回 ''。

7.4.2 获取文章列表

app.get('/api/blogs', (req, res) => {
  db.all(
    `SELECT 
      blogs.id,
      blogs.title,
      blogs.summary,
      blogs.tags,
      blogs.view_count,
      blogs.created_at,
      users.username
    FROM blogs
    JOIN users ON blogs.user_id = users.id
    ORDER BY blogs.created_at DESC`,
    [],
    (err, rows) => {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      res.json(rows);
    }
  );
});

7.4.3 获取单篇文章

app.get('/api/blogs/:id', (req, res) => {
  const blogId = req.params.id;
  
  // 先获取文章
  db.get(
    `SELECT blogs.*, users.username
     FROM blogs
     JOIN users ON blogs.user_id = users.id
     WHERE blogs.id = ?`,
    [blogId],
    (err, blog) => {
      if (err) {
        return res.status(500).json({ error: err.message });
      }
      
      if (!blog) {
        return res.status(404).json({ error: '文章不存在' });
      }
      
      // 浏览次数 +1
      db.run(
        'UPDATE blogs SET view_count = view_count + 1 WHERE id = ?',
        [blogId]
      );
      
      res.json(blog);
    }
  );
});

这里有两个数据库操作：1）查询文章详情；2）更新浏览次数。这两个操作不在同一个事务中，所以浏览次数更新失败不会影响文章查询。

如果你担心并发问题（两个用户同时读同一篇文章，view_count 只加了 1），可以用事务。但对于个人网站，这种并发概率极低，不需要过度优化。

7.4.4 更新文章

app.put('/api/blogs/:id', (req, res) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: '请先登录' });
  }
  
  const { title, content, summary, tags } = req.body;
  const blogId = req.params.id;
  
  // 检查是否是作者
  db.get(
    'SELECT user_id FROM blogs WHERE id = ?',
    [blogId],
    (err, blog) => {
      if (err) return res.status(500).json({ error: err.message });
      if (!blog) return res.status(404).json({ error: '文章不存在' });
      if (blog.user_id !== req.session.userId) {
        return res.status(403).json({ error: '只能修改自己的文章' });
      }
      
      db.run(
        `UPDATE blogs 
         SET title = ?, content = ?, summary = ?, tags = ?, updated_at = CURRENT_TIMESTAMP
         WHERE id = ?`,
        [title, content, summary, tags, blogId],
        function(err) {
          if (err) return res.status(500).json({ error: err.message });
          res.json({ message: '更新成功' });
        }
      );
    }
  );
});

7.6 Markdown 渲染——marked.js

7.6.1 为什么不在后端渲染 Markdown？

有两个选择：

  • 后端渲染：服务器把 Markdown 转成 HTML，前端直接显示
  • 前端渲染：前端拿到 Markdown 文本，用 JavaScript 转成 HTML

我们选择前端渲染，因为：

  • 减轻服务器负担（服务器不用做 Markdown 解析）
  • 可以配合代码高亮库（Prism.js）实现客户端代码高亮
  • 如果需要编辑文章，前端可以直接显示原始 Markdown
  • 支持实时预览（输入 Markdown 时即时看到渲染效果）

🤔 后端渲染有什么优势？

虽然我们用前端渲染，但了解后端渲染的优势也很重要：

  • SEO：搜索引擎爬虫看到的已经是渲染好的 HTML，对 SEO 更友好
  • 首屏速度：浏览器直接收到 HTML，不需要等待 JS 执行
  • 安全性：后端可以控制渲染过程，过滤危险标签

对于需要 SEO 的公开博客，可以用"同构渲染"（Isomorphic Rendering）——首次访问时后端渲染，后续交互用前端渲染。Next.js、Nuxt.js 等框架就是做这个的。

7.6.2 使用 marked.js

步骤
在 blog-detail.html 中引入 marked.js（通过 CDN）： 
<script src="https://cdn.jsdelivr.net/npm/marked/marked.min.js"></script>

使用：

// 把 Markdown 文本转成 HTML
document.getElementById('content').innerHTML = marked.parse(blog.content);

marked.js 的解析流程：1）词法分析（把 Markdown 文本切成 token）；2）语法分析（把 token 转成 HTML AST）；3）渲染（把 AST 转成 HTML 字符串）。

7.6.3 XSS 防护——渲染后的 HTML 也要安全

marked.js 默认会过滤一些危险标签，但不够严格。更好的做法是用 DOMPurify：

<script src="https://cdn.jsdelivr.net/npm/dompurify@3.0.1/dist/purify.min.js"></script>

// 先渲染 Markdown，再净化 HTML
const rawHtml = marked.parse(blog.content);
const safeHtml = DOMPurify.sanitize(rawHtml);
document.getElementById('content').innerHTML = safeHtml;

🤔 为什么 Markdown 也可能有 XSS？

Markdown 允许嵌入 HTML。如果用户输入：

<script>alert('xss')</script>

marked.js 可能会原样保留这段 HTML（取决于配置）。DOMPurify 会删除所有 script 标签和危险属性（如 onclick、onerror），确保最终渲染的 HTML 是安全的。

另一个 XSS 向量是图片：

![x](" onerror="alert('xss'))

DOMPurify 会移除 onerror 属性，阻止这个攻击。

7.6.4 代码高亮——Prism.js

marked.js 渲染代码块时会生成 <pre><code class="language-js">...</code></pre>。要给代码块加语法高亮，需要 Prism.js：

<!-- 引入 Prism.js CSS 和 JS -->
<link href="https://cdn.jsdelivr.net/npm/prismjs@1.29.0/themes/prism-tomorrow.min.css" rel="stylesheet">
<script src="https://cdn.jsdelivr.net/npm/prismjs@1.29.0/prism.min.js"></script>

// 页面加载后调用
Prism.highlightAll();

Prism.js 会根据 code 标签的 class（如 language-js）来应用对应语言的语法高亮规则。

7.7 博客详情页

详情页的核心功能：

  • 从 URL 参数获取文章 ID
  • 请求 /api/blogs/:id 获取文章数据
  • 渲染标题、作者、时间、浏览数
  • 用 marked.js + DOMPurify 渲染正文
  • 用 Prism.js 高亮代码块
// 从 URL 获取 id 参数
const urlParams = new URLSearchParams(location.search);
const blogId = urlParams.get('id');

async function loadBlog() {
  const res = await fetch(`/api/blogs/${blogId}`);
  
  if (!res.ok) {
    document.getElementById('content').innerHTML = '<p>文章不存在</p>';
    return;
  }
  
  const blog = await res.json();
  
  document.getElementById('title').textContent = blog.title;
  document.getElementById('meta').textContent = 
    `${blog.username} · ${new Date(blog.created_at).toLocaleDateString('zh-CN')} · ${blog.view_count} 阅读`;
  
  // 渲染 Markdown
  const rawHtml = marked.parse(blog.content);
  const safeHtml = DOMPurify.sanitize(rawHtml);
  document.getElementById('content').innerHTML = safeHtml;
  
  // 代码高亮
  Prism.highlightAll();
}

7.8 编辑文章功能

编辑页面和发布页面可以共用同一个表单，区别是：

  • 编辑时预填充已有数据
  • 调用 PUT /api/blogs/:id 而不是 POST /api/blogs
  • 只有作者能看到"编辑"按钮
// 页面加载时，如果是编辑模式，获取文章数据预填充表单
async function init() {
  const urlParams = new URLSearchParams(location.search);
  const editId = urlParams.get('edit');
  
  if (editId) {
    const res = await fetch(`/api/blogs/${editId}`);
    const blog = await res.json();
    
    document.getElementById('title').value = blog.title;
    document.getElementById('content').value = blog.content;
    document.getElementById('summary').value = blog.summary;
    document.getElementById('tags').value = blog.tags;
    
    // 修改表单提交逻辑为 PUT
    document.getElementById('form').onsubmit = () => updateBlog(editId);
  }
}

7.9 知识点回顾

📚 本章核心概念回顾

  • Markdown：轻量级标记语言，用纯文本语法生成 HTML。
  • marked.js：JavaScript Markdown 解析器，把 Markdown 转成 HTML。
  • DOMPurify：HTML 净化库，移除危险标签和属性，防止 XSS。
  • Prism.js：代码语法高亮库，支持数百种编程语言。
  • 前端渲染 vs 后端渲染：前端渲染减轻服务器负担，后端渲染利于 SEO。
  • URL 参数：URLSearchParams 用于解析 ?id=123 形式的参数。
  • view_count：非关键数据更新，不需要事务，可以用异步更新。

📝 练习时间

  3. 在博客详情页添加"编辑"按钮，但只有作者能看到。
  4. 实现标签筛选功能：点击某个标签，只显示该标签的文章。
  5. 给博客列表页添加分页功能（每页 10 条，用 LIMIT 和 OFFSET）。

7.10 常见问题急救站

问题现象原因解决方案
Markdown 没渲染 页面显示原始 Markdown 文本 marked.js 没加载或调用方式错误 检查 CDN 链接；用 marked.parse() 而不是 marked()
代码块没高亮 代码块是灰色背景，没有语法高亮 没引入 Prism.js 或 CSS，或没调用 highlightAll() 引入 Prism.js 和对应的主题 CSS；页面加载后调用 highlightAll()
文章列表加载慢 列表页半天没内容 文章太多或 N+1 查询 加 LIMIT 分页；检查是否有循环查询
图片不显示 Markdown 里的图片是裂图 图片路径错误或图片不存在 检查图片 URL 是否正确；相对路径 vs 绝对路径
XSS 攻击成功 输入的 script 被执行了 没用 DOMPurify 或配置不当 marked.parse 后用 DOMPurify.sanitize 净化
浏览数不准确 刷新页面 view_count 不增加 浏览器缓存了 API 响应 API 加缓存控制头；或前端加时间戳参数

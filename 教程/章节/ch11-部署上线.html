第十一章：部署上线——把网站放到公网上

本章目标：把本地开发的网站部署到云服务器，让全世界都能访问。你将理解 Linux 服务器、进程管理、反向代理和 HTTPS。

学完后你能做什么：有一个可以通过域名访问的真实网站，服务器重启后自动恢复。

11.1 部署概述

部署就是把你的代码放到一台"一直开机的电脑"（服务器）上运行。这台电脑有公网 IP，所以任何人都能访问。

11.1.1 需要准备什么？

  • 云服务器：阿里云、腾讯云、AWS、DigitalOcean、Vultr 等
  • 域名：比如 jiguang.com（可选，但推荐）
  • Node.js：服务器上要安装和开发环境相同的 Node 版本
  • Git：用于代码同步

11.1.2 部署流程概览

1. 购买云服务器 → 获取公网 IP
2. 连接服务器 → 安装 Node.js
3. 上传代码 → Git clone 或 scp
4. 安装依赖 → npm install
5. 启动服务 → PM2 进程守护
6. 配置 Nginx → 反向代理 + 静态文件
7. 申请域名 → DNS 解析到服务器 IP
8. 配置 HTTPS → Let's Encrypt 证书

📖 故事时间：云计算的演进

2006 年，Amazon 推出了 EC2（Elastic Compute Cloud）——世界上第一个商业化的云服务器服务。在此之前，如果你想托管一个网站，需要：

  1. 购买物理服务器（几千到几万美元）
  2. 租用机房机架（每月几百美元）
  3. 自己安装操作系统、配置网络
  4. 服务器坏了要自己修

EC2 的创新在于"虚拟化"——一台物理服务器可以虚拟出多台云服务器，按需分配资源。用户只需要在网页上点击几下，就能在几分钟内获得一台可用的服务器。按小时计费，用多少付多少。

这个模式彻底改变了 IT 行业。今天，个人开发者可以用几美元/月的价格租用一台云服务器，而在 2005 年，同样的服务需要几千美元的前期投入。

中国的阿里云（2009）、腾讯云（2010）后来也推出了类似服务，价格更便宜，对国内用户更友好。

11.2 服务器环境准备

11.2.1 连接服务器

通过 SSH 连接到云服务器：

ssh root@你的服务器IP

第一次连接时会提示验证服务器指纹（fingerprint），输入 yes 确认。

11.2.2 安装 Node.js

# Ubuntu/Debian
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# 验证
node --version  # v20.x.x
npm --version   # 10.x.x

NodeSource 是一个第三方仓库，提供比 Ubuntu 官方仓库更新的 Node.js 版本。Ubuntu 官方仓库的 Node 版本通常落后 2-3 年。

🤔 为什么不用 nvm 安装 Node？

nvm（Node Version Manager）是开发环境中管理 Node 版本的好工具，可以切换不同版本。但在生产服务器上，简单稳定更重要——直接用系统包管理器安装一个固定版本，不需要 nvm 的灵活性。

如果你确实需要多版本管理（比如同时运行 Node 16 和 Node 20 的项目），可以用 nvm 或 n（另一个版本管理器）。

11.2.3 安装 Git

sudo apt update
sudo apt install git

git config --global user.name "你的名字"
git config --global user.email "你的邮箱"

11.3 上传代码

11.3.1 用 Git 克隆（推荐）

git clone https://github.com/你的用户名/my-website.git
cd my-website
npm install

这种方式的好处：以后更新代码只需要 git pull。

11.3.2 或者用 scp 上传

scp -r ~/Desktop/my-website root@你的服务器IP:/var/www/

scp（Secure Copy）通过 SSH 加密传输文件。-r 表示递归复制目录。

11.3.3 生产环境配置

生产环境需要一些开发环境没有的设置：

# .env 文件（不要提交到 Git！）
PORT=3000
SESSION_SECRET=your-random-secret-key-here
NODE_ENV=production
npm install dotenv

// server.js 顶部
require('dotenv').config();

const PORT = process.env.PORT || 3000;
const SESSION_SECRET = process.env.SESSION_SECRET || 'fallback-secret';
⚠️ 安全警告
.env 文件包含敏感信息，必须加入 .gitignore： 
echo ".env" >> .gitignore
echo "node_modules/" >> .gitignore
echo "database.sqlite" >> .gitignore
echo "sessions/" >> .gitignore
echo "uploads/" >> .gitignore

11.4 进程守护——PM2

直接用 node server.js 启动的问题是：SSH 断开或服务器重启后，进程会停止。而且如果代码抛出未捕获的异常，进程会崩溃。

PM2 是 Node.js 的进程管理器：崩溃自动重启、开机自动启动、日志管理、负载均衡。

步骤
sudo npm install -g pm2

# 启动
pm2 start server.js --name "my-website"

# 查看状态
pm2 status
pm2 logs my-website

# 重启
pm2 restart my-website

# 停止
pm2 stop my-website

# 开机自启
pm2 startup
pm2 save

pm2 startup 会生成一个系统服务脚本，让 PM2 在系统启动时自动运行。pm2 save 保存当前进程列表，下次启动时自动恢复。

📖 故事时间：进程管理器的发展

早期的 Node.js 部署用 nohup（"no hang up"）命令：nohup node server.js &。这能让进程在 SSH 断开后继续运行，但没有自动重启功能。

2013 年，Alexandre Strzelewicz 创立了 PM2，目标是"让 Node.js 应用的部署和监控变得简单"。PM2 的名字来源于 "Process Manager 2"（第二版）。

PM2 的一个杀手级功能是"集群模式"（Cluster Mode）：它可以启动多个 Node.js 进程，利用多核 CPU。对于高流量网站，这能显著提升性能。

另一个流行选择是 systemd——Linux 系统自带的进程管理器。很多大公司更喜欢 systemd，因为它不需要额外的依赖。但 PM2 对 Node.js 更友好，有专门的日志格式和监控面板。

11.5 Nginx 反向代理

为什么要用 Nginx？

  • Node.js 直接暴露端口不够安全
  • Nginx 可以处理静态文件，减轻 Node 负担
  • Nginx 可以做负载均衡、SSL、压缩、缓存
  • Nginx 可以一个端口服务多个网站（虚拟主机）

11.5.1 安装 Nginx

sudo apt install nginx

# 验证
sudo systemctl status nginx

11.5.2 配置反向代理

# 创建配置文件
sudo nano /etc/nginx/sites-available/my-website

server {
    listen 80;
    server_name www.jiguang.com jiguang.com;

    # 静态文件直接由 Nginx 处理
    location /static/ {
        alias /var/www/my-website/public/;
        expires 30d;
    }

    # 其他请求转发到 Node.js
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_cache_bypass $http_upgrade;
    }
}
# 启用配置
sudo ln -s /etc/nginx/sites-available/my-website /etc/nginx/sites-enabled/
sudo nginx -t  # 测试配置
sudo systemctl reload nginx  # 重载配置

🤔 为什么需要 X-Forwarded-For？

当 Nginx 做反向代理时，Node.js 看到的请求 IP 是 127.0.0.1（本地），而不是用户的真实 IP。X-Forwarded-For 头记录了用户的真实 IP，经过的每一层代理都会追加自己的 IP。

如果你的应用需要记录访问者 IP（如防刷、日志分析），需要读取 X-Forwarded-For 而不是 req.ip。

11.6 HTTPS——SSL 证书

HTTPS 加密了浏览器和服务器之间的通信，防止中间人窃听和篡改。没有 HTTPS 的网站会被浏览器标记为"不安全"。

📖 故事时间：Let's Encrypt 的革命

2014 年之前，SSL 证书需要购买，每年几十到几百美元。小型网站和个人开发者往往不用 HTTPS——成本太高。

2014 年，Mozilla、Cisco、Akamai、IdenTrust 等公司联合发起了 Let's Encrypt 项目，目标是"让互联网上的每一台服务器都能免费获得 SSL 证书"。

2016 年，Let's Encrypt 正式开放。它用 ACME 协议自动验证域名所有权、签发证书、续期证书。整个过程完全自动化，不需要人工干预。

Let's Encrypt 的影响是巨大的：HTTPS 使用率从 2014 年的 30% 飙升到 2024 年的 95% 以上。今天，不用 HTTPS 的网站反而是少数。

Let's Encrypt 的证书有效期只有 90 天（商业证书通常是 1-2 年），这是故意的——迫使用户实现自动化续期，减少因遗忘续期导致的安全问题。

11.6.1 申请 Let's Encrypt 证书

sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d www.jiguang.com -d jiguang.com

certbot 会自动：验证域名所有权 → 申请证书 → 修改 Nginx 配置 → 启用 HTTPS。

11.6.2 自动续期

Let's Encrypt 证书有效期 90 天，需要自动续期：

sudo certbot renew --dry-run  # 测试续期

certbot 安装时会自动创建一个 cron 任务，每天检查证书是否需要续期。

11.7 域名和 DNS

11.7.1 购买域名

常见的域名注册商：阿里云、腾讯云、GoDaddy、Namecheap。一个 .com 域名通常 60-100 元/年。

11.7.2 DNS 解析

在域名管理后台添加 A 记录：

记录类型主机记录记录值说明
A@你的服务器IP根域名 jiguang.com
Awww你的服务器IPwww.jiguang.com

A 记录把域名映射到 IPv4 地址。AAAA 记录用于 IPv6。

DNS 传播需要时间（通常几分钟到几小时），因为 DNS 记录会缓存在各地的 DNS 服务器上。

11.8 完整部署检查清单

🎯 部署前逐一检查

  • □ 云服务器已购买并可以 SSH 连接
  • □ 服务器上安装了 Node.js v20+
  • □ 代码已上传到服务器（Git clone 或 scp）
  • □ npm install 成功安装依赖
  • □ .env 文件已创建，包含 SESSION_SECRET
  • □ .env 在 .gitignore 中
  • □ PM2 启动了网站进程
  • □ Nginx 配置完成，80 端口可访问
  • □ 域名解析到服务器 IP
  • □ HTTPS 证书已申请并生效
  • □ 网站可以通过 https://你的域名 访问
  • □ 重启服务器后网站自动恢复（pm2 startup）

11.9 知识点回顾

📚 本章核心概念回顾

  • SSH：Secure Shell，加密远程登录协议。
  • PM2：Node.js 进程管理器，支持自动重启、开机自启、日志管理。
  • Nginx：高性能 Web 服务器，做反向代理、静态文件服务、负载均衡。
  • 反向代理：客户端 → Nginx → Node.js。Nginx 对外暴露，Node.js 只监听本地。
  • Let's Encrypt：免费的 SSL 证书颁发机构，90 天有效期，自动续期。
  • HTTPS：HTTP + SSL/TLS 加密，防止中间人攻击。
  • DNS A 记录：域名 → IPv4 地址的映射。
  • .env：环境变量文件，存储敏感配置，不提交到 Git。
  • X-Forwarded-For：反向代理时传递原始客户端 IP 的 HTTP 头。

📝 练习时间

  1. 用 Certbot 申请证书后，检查 Nginx 配置文件的变化（多了 ssl_certificate 等指令）。
  2. 配置 HTTP 自动跳转到 HTTPS（301 重定向）。
  3. 添加 gzip 压缩：Nginx 配置 gzip on，减少传输大小。
  4. 配置 Nginx 缓存静态文件：图片/CSS/JS 缓存 30 天。
  5. 用 Docker 容器化部署：Dockerfile + docker-compose.yml。
  6. 配置 CI/CD：GitHub Actions 自动部署到服务器。

11.10 常见问题急救站

问题现象原因解决方案
端口被占用 node server.js 报错 EADDRINUSE 3000 端口已有进程占用 netstat -tlnp | grep 3000 查找并 kill；或换端口
防火墙阻止 浏览器访问超时 云服务器安全组没开放 80/443 在控制台添加安全组规则（入站 80, 443, 22）
PM2 进程不启动 重启后网站无法访问 没执行 pm2 save pm2 save 保存当前进程列表；pm2 startup 配置开机自启
静态文件 404 CSS/JS 文件加载失败 Nginx 没配置静态文件路径 检查 location /static/ 的 alias 路径；检查文件权限
证书过期 浏览器警告"不安全" Let's Encrypt 证书没续期 certbot renew；检查 cron 任务是否正常
域名不解析 ping 域名不通 DNS 没配置或没传播 检查 DNS A 记录；用 dig 或 nslookup 检查；等待传播
HTTPS Mixed Content 页面部分资源不加载 HTTPS 页面引用了 HTTP 资源 所有资源改为 https:// 或 //（协议相对 URL）
